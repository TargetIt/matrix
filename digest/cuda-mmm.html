从最笨的矩阵乘法，走到接近 cuBLAS

原文是一篇优化工作日志。它的价值不是只给结论，而是把“为什么慢、改哪里、提升多少”按 kernel 版本铺开。下面保留这条主线，并在每一段后面加中文解释。

原始网页：How to Optimize a CUDA Matmul Kernel for cuBLAS-like Performance

原文线索

原文主题：用 CUDA 手写 SGEMM，从 Naive kernel 逐步加入 coalescing、shared memory、block tiling、vectorized access、autotuning、warptiling，最后接近 cuBLAS 的 FP32 性能。
关键数据：原文 A6000 示例从 Naive 约 309 GFLOP/s 开始，最后 warptiling 版本约 21.8 TFLOP/s，接近 cuBLAS 的 93% 以上。
先不要被这些数字吓到。你只要记住：同一台 GPU，算法数学公式没变，只是数据的“排队方式、搬运方式、复用方式”变了，速度就能差几十倍。

原文结构脉络

开篇问题为什么矩阵乘法对深度学习这么重要，以及手写 CUDA SGEMM 离 cuBLAS 有多远。
Kernel 1朴素实现：一个线程算一个 C 元素，暴露最基础的 GMEM 访问问题。
Kernel 2Global memory coalescing：让同一 warp 的访问更连续。
Kernel 3Shared memory cache-blocking：把 A/B tile 先搬到 block 共享的小缓存。
Kernel 4-6Block tiling、thread tiling、vectorized access：逐步提高复用率和访存效率。
后续版本Autotuning 与 warptiling：把参数搜索和 warp 级分工纳入优化。
这篇原文最适合按“版本升级日志”来读。每一版都不是推翻上一版，而是在上一版的瓶颈上再修一层：先让访问连续，再让数据复用，再让线程负责更多输出，再让 warp 层级更合理。

第一层：Naive kernel 到底在做什么

最朴素的做法是：一个线程负责输出矩阵 C 的一个元素。这个线程拿 A 的一整行、B 的一整列，沿着 K 方向一个个相乘再加起来。

C[row, col] = sum over k: A[row, k] * B[k, col]
这里的关键不是公式，而是访问路径。一个线程为了算一个 C 元素，会扫过 A 的一行和 B 的一列。如果很多线程各扫各的，就可能重复读同一批 A/B 数据。数学上很清楚，硬件上却很浪费。
像一群人同时做饭，每个人都去仓库拿同一袋米。饭没问题，但路上浪费太多时间。

第二层：coalescing 是让 32 个线程排好队拿东西

GPU 不是一个线程一个线程地随便跑。常见执行单位是 warp，一组 32 个线程一起执行。如果这 32 个线程访问连续地址，硬件可以把多次小访问合并成少数几次大事务。

Coalescing 的中文直觉是“合并访存”。不是让算法少算，而是让同一 warp 里的线程访问地址尽量连续。连续访问像整箱搬货；离散访问像每次只拿一件，还要跑很多趟。
坏访问相邻线程访问相距很远的地址，硬件无法合并，带宽浪费严重。
好访问相邻线程访问连续 float，32 个线程可以接近一次 128B 事务。

第三层：Shared Memory 是 block 的小货架

当一块 A/B 数据会被同一个 block 里的多个线程反复使用时，就值得先把它从 Global Memory 搬到 Shared Memory。

SMEM 的作用不是“凭空变快”，而是把远处仓库的数据搬到近处货架。搬一次以后，block 内线程可以反复用它。矩阵乘法非常适合这样做，因为 A 的一个 tile 会服务 C 的多列，B 的一个 tile 会服务 C 的多行。
for each K tile:
  load A tile and B tile from GMEM to SMEM
  __syncthreads()
  compute many FMA using SMEM data
  __syncthreads()
两个同步点都很重要：第一次保证货架装满了再取；第二次保证大家用完了再换下一批货。

第四层：tiling 是把大矩阵切成小任务

Block tiling、thread tiling、warp tiling 都是在回答同一个问题：这块 C 由谁负责？A/B 的哪一块会被复用？中间结果放哪里？

Block tile

一个 thread block 负责 C 的一块矩形区域。

Warp tile

block 内的不同 warp 再分 C 的子块。

Thread tile

一个线程不只算一个 C，而是算一小片 C。

Register tile

线程自己的多个 accumulator 放在寄存器里。

切块不是为了好看，而是为了“同一批食材尽量多做几道菜”。切得合适，搬一次 A/B，就能贡献很多个 C。

第五层：读完这篇后应该带走什么

  • Naive 慢，主要是数据访问太散、复用太低。
  • Coalescing 先把 global memory 访问排整齐。
  • SMEM 把可复用数据搬到更近的位置。
  • Thread/warp tiling 把中间结果留在寄存器里，多算少搬。
  • 最后的性能来自一整套配合，不是某一个技巧单独起作用。
