这个目录的目标不是再放三个链接，而是把三篇资料重新组织成一条清楚的学习路线：先从“矩阵乘法到底在干什么”开始，再讲“GPU 为什么会被数据搬运拖慢”，最后讲“CUTLASS 为什么要把 GEMM 拆成那么多层”。新增中文解释都用“导读补充”“给非专业读者”“深入一点”“读完自测”标记出来。

## 先把整件事讲成一句人话

> **公式**：SGEMM 的本质：很多线程一起算很多个点积。优化的本质：同样的数据尽量少从远处搬，多在近处重复使用。

> **给非专业读者**
>
> 想象一个小区食堂要做 1000 份同样的菜。最笨的方法是每个厨师每炒一份都跑去大仓库拿米、油、盐。聪明的方法是先把一批米、油、盐搬到灶台旁的小货架，再让一组厨师反复使用。GPU 里的 Global Memory 像大仓库，Shared Memory 像灶台旁的小货架，Registers 像厨师手里的小碗。SGEMM 优化就是把“跑仓库”的次数降下来。

> **导读补充**
>
> 从第一性原理看，矩阵乘法没有魔法。每个 C 元素都等于 A 的一行和 B 的一列相乘再求和。真正的问题是：一个大矩阵有太多行和列，很多线程会反复需要同一批 A/B 数据。如果每个线程都从最慢的 Global Memory 重读，就算 GPU 有很强的计算能力，也会因为“数据供应跟不上”而空转。

## 三篇文章在学习路线中的位置

- **第一篇：CUDA MMM Worklog**：负责把问题从零讲起来。它像一份实验笔记：先写最笨的 kernel，然后每次只改一个主要瓶颈，观察速度怎么变。
- **第二篇：SGEMM Practice**：负责把概念落到代码。它把常见优化写成 7 个 CUDA kernel，适合看 threadIdx 怎么映射、tile 怎么搬、寄存器怎么存。
- **第三篇：CUTLASS**：负责把经验上升成框架。它把 thread block、warp、thread、instruction 四层组织成模板参数，说明高性能 GEMM 可以系统化生成。

> **导读补充**
>
> 建议按这个顺序读，不要反过来。直接读 CUTLASS 会像一上来就看汽车发动机图纸；先读 Worklog 和 Practice，等你知道“为什么要切块、为什么要共享内存、为什么要寄存器 tile”，再读 CUTLASS，模板参数才会变成有意义的名字。

## 用一个 2×2 的小矩阵建立直觉

假设：

```text
A = [[1, 2],
     [3, 4]]

B = [[5, 6],
     [7, 8]]
```

那么 C 的左上角元素是 A 第一行和 B 第一列的点积：

```text
C[0,0] = 1×5 + 2×7 = 19
```

四个输出分别是：

| 输出位置 | 计算过程 | 结果 |
| --- | --- | --- |
| C[0,0] | 1×5 + 2×7 | 19 |
| C[0,1] | 1×6 + 2×8 | 22 |
| C[1,0] | 3×5 + 4×7 | 43 |
| C[1,1] | 3×6 + 4×8 | 50 |

> **导读补充**
>
> 这个小例子已经包含 SGEMM 的全部本质：A 的同一行会被拿去算多个列，B 的同一列会被拿去算多个行。比如 A 的第一行 `[1,2]` 同时参与 C[0,0] 和 C[0,1]；B 的第一列 `[5,7]` 同时参与 C[0,0] 和 C[1,0]。优化的核心就是抓住这种“重复使用”，不要每次都重新从最慢的地方拿。
