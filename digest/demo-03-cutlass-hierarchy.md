# CUTLASS 分层 GEMM 结构：从 Thread Block 到 Thread Tile 的手把手推导 (0-9 整数矩阵)

> 本文基于 [NVIDIA 官方博客](https://developer.nvidia.com/blog/cutlass-linear-algebra-cuda/)，以 M=N=K=16 的 demo 为例，自顶向下拆解 CUTLASS 的三层 GEMM 分块策略。每一个数字均来自配套 Python 验证脚本 `demo-03-cutlass-hierarchy.py`，与代码输出完全一致（0-9 整数矩阵，seed=1，float32）。

## 1. Demo 参数设定

```
问题规模: M = N = K = 16
数据类型: int32 randint(0,10) → float32
随机种子: 1

分块参数:
  Thread Block Tile:  BM = 8, BN = 8, BK = 4
  Warp Tile:          WM = 4, WN = 4
  Thread Tile:        TM = 2, TN = 2

```

## 2. 层次结构总览

CUTLASS 将 GEMM 计算拆成三层嵌套的 tile 结构：

```
C[16×16]
  ├─ Block (0,0): C[0:8, 0:8]     Block (0,1): C[0:8, 8:16]
  └─ Block (1,0): C[8:16, 0:8]    Block (1,1): C[8:16, 8:16]

每个 Block (8×8) 内部:
  ├─ Warp (0,0): 4×4    Warp (0,1): 4×4
  └─ Warp (1,0): 4×4    Warp (1,1): 4×4

每个 Warp (4×4) 内部:
  ├─ Thread (0,0): 2×2   Thread (0,1): 2×2
  └─ Thread (1,0): 2×2   Thread (1,1): 2×2
```

Block 级通过 K 维度上的外层循环（K/BK = 4 次迭代）将 16 维的 K 分批处理。数据流动路径：

```
GMEM (全局内存) ──load──► SMEM (共享内存) ──load──► RF (寄存器) ──FMA──► RF
     A, B, C              A_tile, B_tile           A_frag, B_frag       累加器
```

## 3. Level 1 — Thread Block Tile：聚焦 Block (0,0)

### 如何计算 Grid 大小

Grid 大小由问题规模 M、N 和 Thread Block Tile 大小 BM、BN 共同决定：

```
gridDim = (CEIL_DIV(M, BM), CEIL_DIV(N, BN)) = (16/8, 16/8) = (2, 2)
```

- `gridDim.m = CEIL_DIV(M, BM)` = 沿 M 方向需要 16/8 = 2 个 Block
- `gridDim.n = CEIL_DIV(N, BN)` = 沿 N 方向需要 16/8 = 2 个 Block
- 总计启动 2 × 2 = 4 个 Thread Block，每个 Block 负责一个 8×8 的 C 子块

Block 坐标 (i, j) 负责的子块范围：
- `C[i*BM : (i+1)*BM, j*BN : (j+1)*BN]`

### Thread Block (0,0) 内部结构

```
Thread Block (0,0) 内部结构:
C[0:8, 0:8] (BM=8, BN=8)
├── K=0..3 (BK=4):  加载 A[0:8,0:4]→SMEM, B[0:4,0:8]→SMEM
├── K=4..7 (BK=4):  加载 A[0:8,4:8]→SMEM, B[4:8,0:8]→SMEM
```

## 4. Level 2 — Warp Tile

### Block(0,0) 的 Warp 划分与内部结构

```
Block(0,0) 的 Warp 划分 (2×2 warps):
      ┌──────────┬──────────┐
      │ Warp(0,0)│ Warp(0,1)│
      │ C[0:4,   │ C[0:4,   │
      │    0:4]  │    4:8]  │
      ├──────────┼──────────┤
      │ Warp(1,0)│ Warp(1,1)│
      │ C[4:8,   │ C[4:8,   │
      │    0:4]  │    4:8]  │
      └──────────┴──────────┘

Warp(0,0) 内部 (WM=4, WN=4):
├── [SMEM→RF] 加载 A fragment: 4×1 (或 4×BK 取决于实现)
├── [SMEM→RF] 加载 B fragment: 1×4 (或 BK×4)
└── 4 Threads × 4 elements/thread = 16 elements
    每个 Warp 内线程组织: 2×2 grid
    ├── Thread(0,0): C[0:2, 0:2]
    ├── Thread(0,1): C[0:2, 2:4]
    ├── Thread(1,0): C[2:4, 0:2]
    └── Thread(1,1): C[2:4, 2:4]
```

### 4.1 8×8 Block → 4 个 4×4 Warp

```
         col 0-3                 col 4-7
  ┌───────────────────┬───────────────────┐
  │ Warp (0,0) 4×4    │ Warp (0,1) 4×4    │  row 0-3
  │ C[0:4, 0:4]       │ C[0:4, 4:8]       │
  ├───────────────────┼───────────────────┤
  │ Warp (1,0) 4×4    │ Warp (1,1) 4×4    │  row 4-7
  │ C[4:8, 0:4]       │ C[4:8, 4:8]       │
  └───────────────────┴───────────────────┘
```

每个 Warp 不直接访问全局内存——它从 Block 已加载到共享内存 (SMEM) 的 A_tile (8×4) 和 B_tile (4×8) 中读取自己需要的行/列切片。

**内存流动：SMEM → RF (寄存器)**

## 5. Level 3 — Thread Tile：完整追踪 Thread (0,0)

### Thread(0,0) 内部结构

```
Thread(0,0) in Warp(0,0):
├── 负责: C[0:2, 0:2] = 4 elements (TM=2, TN=2)
├── [SMEM→RF] reg_A[2] ← As[0:2, k]  (A 的一列片段)
├── [SMEM→RF] reg_B[2] ← Bs[k, 0:2]  (B 的一行片段)
└── [RF: FMA] 计算:
    C[0,0] += reg_A[0] * reg_B[0]
    C[0,1] += reg_A[0] * reg_B[1]
    C[1,0] += reg_A[1] * reg_B[0]
    C[1,1] += reg_A[1] * reg_B[1]
    4 次 FMA / thread / k-step
```

### 5.1 Warp 内的 Thread 布局

Warp (0,0)（即 C[0:4, 0:4]）内有 2×2 = 4 个线程：

```
Thread (0,0) → C[0:2, 0:2]
Thread (0,1) → C[0:2, 2:4]
Thread (1,0) → C[2:4, 0:2]
Thread (1,1) → C[2:4, 2:4]
```

每个线程独立计算自己的 2×2 小块，累加 4 次 K 迭代。

**内存流动：SMEM → RF → FMA**

在每个 K 迭代中：
1. Block 级别：`GMEM → SMEM`（加载 A_tile, B_tile 到共享内存，所有线程共享）
2. Thread 级别：`SMEM → RF`（Thread 从 SMEM 中加载自己的 A_frag, B_frag 到寄存器）
3. 计算：`RF → FMA`（在寄存器中执行矩阵乘法）

## 6. Software Pipelining — Ping-Pong 双缓冲

### Pipeline 时间线

```
Pipeline 时间线 (3 stages overlapped):
Stage 1 (最慢): [GMEM→SMEM] 加载下一 tile
Stage 2 (中等): [SMEM→RF]   加载 warp fragment  
Stage 3 (最快): [RF→CUDA]   FMA 计算
                            
时间 →  ┌──── Stage1(k=0) ────┐┌──── Stage1(k=1) ────┐
        │  ┌── Stage2(k=0) ──┐││  ┌── Stage2(k=1) ──┐│
        │  │ ┌ Stage3(k=0) ┐ │││  │ ┌ Stage3(k=1) ┐ ││
        │  │ │  FMA FMA    │ │││  │ │  FMA FMA    │ ││
        ▼  ▼ ▼             ▼ ▼▼  ▼ ▼             ▼ ▼▼
迭代 k:   0                   1                   2
```
Each stage feeds data to the next: Stage1→SMEM, Stage2→RF, Stage3 uses RF for FMA

在真实 CUDA kernel 中，全局内存访问延迟很高（~500 cycles）。CUTLASS 使用双缓冲来重叠计算与访存：

```
时间 ───────────────────────────────────────────►
迭代 0: [GMEM→buf_A 加载 tile 0] [用 tile 0 计算]
迭代 1:                          [GMEM→buf_B 加载 tile 1] [用 tile 1 计算]
迭代 2:                                                   [GMEM→buf_A 加载 tile 2] ...
迭代 3:                                                                           [GMEM→buf_B 加载 tile 3] ...
```

实现要点：
- Block 分配两块等大的 shared memory（buf_A, buf_B）
- 计算 tile k 的同时，异步加载 tile k+1 到另一块 buffer
- `__syncthreads()` 确保加载完成后才开始计算
- 首轮需"预加载"（prologue），末轮需"排空"（drain）

## 7. Epilogue — 块内融合后处理（0-9 整数矩阵）

## 8. WMMA / Tensor Core 抽象

Volta 及更新架构（Turing、Ampere、Hopper）上，最内层的"Thread Tile"被替换为 Warp 级的 Tensor Core 指令：

```
Thread Tile (标量 FMA)  →  WMMA / mma.sync (Tensor Core)
```

上层 Block Tile → Warp Tile 的结构完全不变，只替换最内层的计算原语。这正是 CUTLASS 分层设计的关键优势：**分层抽象使底层计算原语可插拔**。
