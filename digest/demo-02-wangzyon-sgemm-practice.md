## 2. K1: Naive 全局内存实现

### 2.1 核心思想

最直接的方式：每个线程计算 C 的一个元素。线程配置：

- `blockDim = (4, 4)` → 每 block 16 线程
- `gridDim = (4, 4, 1)` → 16 blocks
- 总线程数 = 16 × 16 = 256，恰好覆盖 16×16 的 C

每个线程执行： `C[row][col] = Σ(k=0..15) A[row][k] × B[k][col]`

### 2.2 计算访存比分析

**从第一性原理角度**：SGEMM 是典型的 memory-bound 问题。对于 M=N=K=N 的方阵：

- **计算量:** M×N×K = 16×16×16 = 4096 次 FMA（融合乘加）= 8192 FLOPs
- **全局内存访问（K1 无缓存复用）:**
  - A 读取: 256 threads × 16 elements = 4096 次
  - B 读取: 256 threads × 16 elements = 4096 次
  - C 写入: 256 次
  - 总流量: (4096+4096+256) × 4 bytes = **33,792 bytes**

```
算术强度 (Arithmetic Intensity) = 8192 FLOPs / 33792 bytes ≈ 0.242 FLOPs/byte
```

**0.242 意味着什么？** 每从内存读取 1 byte 数据，只做 0.242 次浮点运算。考虑到现代 GPU 的内存带宽（如 A100: ~2 TB/s）和计算能力（~19.5 TFLOPS），要达到计算饱和需要 ~9.75 FLOPs/byte。当前 0.242 远远不够——**被内存带宽严重限制**。

### 2.3 C[0,0] 的完整计算

线程 block(0,0), thread(0,0) 负责 C[0,0]:

```
C[0,0] = A[0,0]×B[0,0] + A[0,1]×B[1,0] + ... + A[0,15]×B[15,0]
```

### 2.4 K1 的问题

每个线程独立地从全局内存读取所有需要的 A 和 B 元素。**同一元素被重复读取多次**——A[0,0] 被 Block(0,0) 的 16 个线程读取 16 次，还被 Block(0,1)、Block(0,2)、Block(0,3) 的线程读取，总共被读取 16×4 = 64 次（每个需要 A[0,k] 来算 C[0,*] 的线程）。

## 3. K2: 共享内存缓存 (SMEM Caching)

### 3.1 核心思想

**第一性原理**: 如果多个线程需要同一数据，在更近的存储层次缓存它。GPU 的共享内存（Shared Memory）是 block 内所有线程共享的片上 SRAM，延迟 ~20-30 cycles vs 全局内存的 ~300-500 cycles。

**参数:** BM=BN=BK=4, blockDim=(4,4), gridDim=(4,4)

将 K 维分成 4 步（BK=4），每一步：
1. Block 协作加载 4×4 tile 的 A 和 B 到共享内存（只从全局内存读一次）
2. `__syncthreads()` 确保所有数据就绪
3. 16 个线程从共享内存读取数据，计算偏微和

### 3.2 Tile 滑动模式

Block(0,0) 覆盖 C[0:4, 0:4]，K 方向分 4 步滑动：

```
K step 0: A[0:4,0:4] × B[0:4,0:4]   → partial C
K step 1: A[0:4,4:8] × B[4:8,0:4]   → partial C
K step 2: A[0:4,8:12] × B[8:12,0:4]  → partial C
K step 3: A[0:4,12:16] × B[12:16,0:4] → partial C

C[0:4,0:4] = Σ partial
```

### 3.3 详细计算：C[0,0]

**K step 0 (k=0..3):** 共享内存加载 A[0:4, 0:4] 和 B[0:4, 0:4]

A_smem (4×4):
```
[ +0.496714  -0.138264  +0.647689  +1.523030 ]
[ -1.012831  +0.314247  -0.908024  -1.412304 ]
[ -0.013497  -1.057711  +0.822545  -1.220844 ]
[ +0.343618  -1.763040  +0.324084  -0.385082 ]
```

B_smem (4×4):
```
[ +1.266911  -0.707669  +0.443819  +0.774634 ]
[ +1.163164  +0.010233  -0.981509  +0.462103 ]
[ +0.280992  -0.622699  -0.208122  -0.493001 ]
[ -0.020902  +0.117327  +1.277665  -0.591571 ]
```

偏微 C[0:4, 0:4] (k=0..3):
```
[ +0.618630  -0.577547  +2.167282  -0.899413 ]
[ -1.143274  +1.119690  -2.373421  +0.643777 ]
[ -0.990745  -0.656709  -0.698857  -0.182527 ]
[ -1.516257  -0.508197  +1.323489  -0.480499 ]
```

其中 C[0,0] 的偏微 = A[0,0]×B[0,0] + A[0,1]×B[1,0] + A[0,2]×B[2,0] + A[0,3]×B[3,0]
= 0.496714×1.266911 + (-0.138264)×1.163164 + 0.647689×0.280992 + 1.523030×(-0.020902)
= 0.629293 + (-0.160824) + 0.181995 + (-0.031834)
= **+0.618630**

**K step 1 (k=4..7):** 偏微 C[0,0] = A[0,4]×B[4,0] + A[0,5]×B[5,0] + A[0,6]×B[6,0] + A[0,7]×B[7,0]
= (-0.234153)×0.096996 + (-0.234137)×(-0.825497) + 1.579213×0.857660 + 0.767435×0.097676
= (-0.022712) + 0.193279 + 1.354427 + 0.074960
= **+1.599954**

**K step 2 (k=8..11):** 偏微 C[0,0] = A[0,8]×B[8,0] + A[0,9]×B[9,0] + A[0,10]×B[10,0] + A[0,11]×B[11,0]
= (-0.469474)×(-0.759133) + 0.542560×(-1.594428) + (-0.463418)×1.551152 + (-0.465730)×(-0.269407)
= 0.356393 + (-0.865073) + (-0.718831) + 0.125471
= **-1.102040**

**K step 3 (k=12..15):** 偏微 C[0,0] = A[0,12]×B[12,0] + A[0,13]×B[13,0] + A[0,14]×B[14,0] + A[0,15]×B[15,0]
= 0.241962×(-0.513867) + (-1.913280)×(-0.112328) + (-1.724918)×(-0.127918) + (-0.562288)×(-1.037246)

= **+0.894457**

**最终 C[0,0]:** 0.618630 + 1.599954 + (-1.102040) + 0.894457 = **+2.011001** ✓

### 3.4 访存分析

K2 的全局内存读取:

- A: 16 blocks × 16 elements × 4 ksteps = 1024 次
- B: 16 blocks × 16 elements × 4 ksteps = 1024 次
- C 写入: 256 次
- 总流量: (1024+1024+256) × 4 = **9,216 bytes**

```
AI(K2) = 8192 / 9216 ≈ 0.889 FLOPs/byte  (K1 的 3.67 倍)
```

**为什么减少了？** 在 K1 中，A[0,0] 被 64 个线程各自从全局内存读取。在 K2 中，A[0,0] 被 block 的所有 16 个线程通过一次全局加载 + 共享内存广播共享。虽然不同 block 之间仍有重复，但 block 内部的重复被消除了。A[0,0] 从被读 64 次降到 4 次（4 个 block 各加载一次）。

## 4. K3: 一维线程 Tile (1D Thread Tile)

### 4.1 核心思想

**第一性原理**: 在 K2 中每个线程只负责 1 个 C 元素，每次 FMA 都要从共享内存读取。如果让每个线程负责多个 C 元素，就可以复用从共享内存加载的寄存器数据——一次加载，多次使用。

**参数:** BM=8, BN=8, BK=4, TM=4. blockDim=(8,2), gridDim=(2,2)

- blockDim.x=8 (N方向线程), blockDim.y=2 (M方向线程)
- 每个 M 方向线程计算 TM=4 个连续 C 元素（沿 M 方向展开）

**线程数:** 2×2×8×2 = 64 线程（vs K2 的 256）
**每线程工作量:** 4 个 C 元素（vs K1/K2 的 1 个）

### 4.2 线程映射

Block(0,0) 覆盖 C[0:8, 0:8] = 64 个元素:

| 线程 | 负责的 C 元素 |
|------|-------------|
| (ty=0, tx=0) | C[0,0], C[1,0], C[2,0], C[3,0] |
| (ty=1, tx=0) | C[4,0], C[5,0], C[6,0], C[7,0] |
| (ty=0, tx=1) | C[0,1], C[1,1], C[2,1], C[3,1] |
| (ty=1, tx=1) | C[4,1], C[5,1], C[6,1], C[7,1] |
| ... | ... |
| (ty=0, tx=7) | C[0,7], C[1,7], C[2,7], C[3,7] |
| (ty=1, tx=7) | C[4,7], C[5,7], C[6,7], C[7,7] |

### 4.3 寄存器级操作：thread(ty=0, tx=0)

该线程持有 4 行 A 数据和 1 列 B 数据（对于 kt=0）：

**reg_A (4×4) = A[0:4, 0:4]:**
```
reg_A[0] = [ +0.496714  -0.138264  +0.647689  +1.523030 ]
reg_A[1] = [ -1.012831  +0.314247  -0.908024  -1.412304 ]
reg_A[2] = [ -0.013497  -1.057711  +0.822545  -1.220844 ]
reg_A[3] = [ +0.343618  -1.763040  +0.324084  -0.385082 ]
```

**reg_B (4×1) = B[0:4, 0]:**
```
reg_B[0] = +1.266911
reg_B[1] = +1.163164
reg_B[2] = +0.280992
reg_B[3] = -0.020902
```

**FMA 计算（kt=0 偏微）：**

C[0,0] 偏微:
```
reg_A[0][0]×reg_B[0] = +0.496714 × +1.266911 = +0.629293
reg_A[0][1]×reg_B[1] = -0.138264 × +1.163164 = -0.160824
reg_A[0][2]×reg_B[2] = +0.647689 × +0.280992 = +0.181995
reg_A[0][3]×reg_B[3] = +1.523030 × -0.020902 = -0.031834
                                          Σ = +0.618630
```

C[1,0] 偏微:
```
reg_A[1][0]×reg_B[0] = -1.012831 × +1.266911 = -1.283167
reg_A[1][1]×reg_B[1] = +0.314247 × +1.163164 = +0.365521
reg_A[1][2]×reg_B[2] = -0.908024 × +0.280992 = -0.255147
reg_A[1][3]×reg_B[3] = -1.412304 × -0.020902 = +0.029519
                                          Σ = -1.143274
```

C[2,0] 偏微 = **-0.990745**（验证于脚本）
C[3,0] 偏微 = **-1.516257**（验证于脚本）

### 4.4 关键：B 列数据的复用

thread(ty=0, tx=0) 将 B[0:4, 0] 加载一次到寄存器，然后与 reg_A 的 4 行分别做点积——**一次加载，4 次使用**。这就是 Thread Tile 复用的本质。

### 4.5 访存分析

- A 读取: 4 blocks × 4 ksteps × 32 = 512 次
- B 读取: 4 blocks × 4 ksteps × 32 = 512 次
- C 写入: 1024 次
- 总流量: (512+512+1024) × 4 = **8,192 bytes**

```
AI(K3) = 8192 / 8192 ≈ 1.000 FLOPs/byte  (K1 的 4.12 倍)
```

## 5. K4: 二维线程 Tile (2D Thread Tile)

### 5.1 核心思想

将 thread tile 从 1D 扩展到 2D：一个线程计算一个 TM×TN = 4×4 = **16 个 C 元素**。

**参数:** BM=8, BN=8, BK=4, TM=4, TN=4. blockDim=(2,2), gridDim=(2,2)

- 每个 block 只有 4 个线程！
- 每个线程负责 4×4=16 个 C 元素
- Block(0,0) 的 4 个线程覆盖 C[0:8, 0:8] = 64 元素

### 5.2 线程块划分

Block(0,0) 覆盖 C[0:8, 0:8]:

```
+-------------------+-------------------+
| Thread(0,0)       | Thread(0,1)       |
| C[0:4, 0:4]       | C[0:4, 4:8]       |
| 16 elements       | 16 elements       |
+-------------------+-------------------+
| Thread(1,0)       | Thread(1,1)       |
| C[4:8, 0:4]       | C[4:8, 4:8]       |
| 16 elements       | 16 elements       |
+-------------------+-------------------+
```

### 5.3 Thread(0,0) 的寄存器操作

持有 reg_A[4][4]（A[0:4, 0:4]）和 reg_B[4][4]（B[0:4, 0:4]），执行 4×4 矩阵乘法得到 16 个偏微值：

```
[ +0.618630  -0.577547  +2.167282  -0.899413 ]    C[0:4,0] 偏微
[ -1.143274  +1.119690  -2.373421  +0.643777 ]    C[0:4,1] 偏微
[ -0.990745  -0.656709  -0.698857  -0.182527 ]    C[0:4,2] 偏微
[ -1.516257  -0.508197  +1.323489  -0.480499 ]    C[0:4,3] 偏微
```

每列是一个独立的内积：reg_A 的行向量与 reg_B 的第 j 列做点积。

### 5.4 寄存器复用倍数

在 K4 中，reg_B[0][0]（=B[0,0]=+1.266911）被用于计算 C[0,0]、C[1,0]、C[2,0]、C[3,0]——**一次加载，4 次使用**。同理 reg_A[0][0] 被用于计算 C[0,0]、C[0,1]、C[0,2]、C[0,3]——也是 **1:4 复用**。

总复用率：每个从共享内存加载的 float 平均用于 **4 次 FMA**（2D tile 下是 TM×TN/BK = 16/4 = 4 倍）。

### 5.5 访存分析

K4 与 K3 相同的数据流（相同 SMEM tile 尺寸），所以 AI 相同：

```
AI(K4) ≈ 1.000 FLOPs/byte
```

但线程数减少（64→16），每个线程的寄存器压力增大，ILP（指令级并行）潜力增加。

## 6. K5: 寄存器缓存 (Register Caching)

### 6.1 核心思想

**第一性原理**: 在 K4 中，每次 FMA 的 operands 来自共享内存。共享内存虽快（~20-30 cycles），但寄存器更快（~0 cycles 延迟，如果编译器优化得当）。

K5 将每个线程需要的 As/Bs 片段显式加载到局部寄存器数组：

```cuda
float reg_A[TM][BK];  // reg_A[4][4] = 16 registers
float reg_B[BK][TN];  // reg_B[4][4] = 16 registers
```

然后对 reg_A 和 reg_B 执行 4×4×4 的 FMA 循环——所有操作数都在寄存器中。

### 6.2 内存层次

```
全局内存 (HBM)          ~300-500 cycles, ~1-2 TB/s
    ↓ 协作加载
共享内存 (SRAM)         ~20-30 cycles, ~10 TB/s (per SM)
    ↓ 寄存器加载
寄存器 (Register File)  ~0 cycles (编译器调度)
    ↓ FMA 指令
计算单元 (CUDA Core)    1 cycle per FMA (warp)
```

**关键洞察**: 数据从全局内存到共享内存，再加载到寄存器。一旦数据在寄存器中，后续 FMA 完全不受内存延迟影响——这是 "register tiling" 的本质。

### 6.3 K5 vs K4 的区别

| 方面 | K4 | K5 |
|------|----|----|
| SMEM 加载 | 隐式（编译器调度） | 不变 |
| FMA 操作数来源 | 共享内存 | 寄存器 |
| 编译器可优化性 | 中等 | 高（显式展开） |
| 计算访存比 | 1.000 | 1.000（数据量相同） |

K5 的改进不在访存比，而在**指令执行效率**——寄存器操作数让编译器更容易做指令重排和延迟隐藏。

## 7. K6: Float4 向量化加载

### 7.1 核心思想

CUDA 的内存总线宽度是 128-bit（32 bytes × 4 = 128 bits）。用 `float`（32-bit）加载每次只用总线容量的 1/4。`float4` 一次加载 128 bits = 4 个 float，充分利用总线。

```cuda
// K5: 4 次 32-bit 加载
smem[0] = global[0];
smem[1] = global[1];
smem[2] = global[2];
smem[3] = global[3];

// K6: 1 次 128-bit 加载
reinterpret_cast<float4*>(smem)[0] = reinterpret_cast<float4*>(global)[0];
```

### 7.2 具体示例

Block(0,0), kt=0 时加载 A[0:8, 0:4] 到共享内存。

用 float4 加载，每行 4 个 float 打包为一次加载：

```
float4(A[0,0], A[0,1], A[0,2], A[0,3])
  = float4(0.496714, -0.138264, 0.647689, 1.523030)

float4(A[1,0], A[1,1], A[1,2], A[1,3])
  = float4(-1.012831, 0.314247, -0.908024, -1.412304)
```

8 行 × 每行 1 次 float4 = **8 次事务**（vs K5 的 32 次）
4 列 B × 每列 1 次 float4 = **8 次事务**（vs K5 的 32 次）

### 7.3 效果

数据量相同，内存事务数减少 4 倍。对于 bandwidth-bound 的 SGEMM，更高的总线利用率直接转化为更短的加载时间。

---

## 8. K7: 双缓冲 (Double Buffering)

### 8.1 核心思想

**第一性原理**: Pipeline 化。加载和计算是两个独立的硬件单元（内存控制器 vs CUDA Cores）。如果让它们串行执行，总时间 = 加载时间 + 计算时间。如果让它们重叠，总时间 ≈ max(加载时间, 计算时间)。

双缓冲使用两套共享内存（PING 和 PONG），交替用于加载和计算。

### 8.2 Ping-Pong 时间线

Block(0,0) 在 4 个 K 步上的执行：

```
Step 0: [LOAD  A[0:8,0:4], B[0:4,0:8]  → PING]    (预加载，无计算)
         __syncthreads()

Step 1: [LOAD  A[0:8,4:8], B[4:8,0:8]  → PONG]    ←── 加载与计算重叠
         [COMPUTE with PING]                        ←── 计算 k=0..3 的偏微
         __syncthreads()

Step 2: [LOAD  A[0:8,8:12], B[8:12,0:8] → PING]   ←── 加载与计算重叠
         [COMPUTE with PONG]                        ←── 计算 k=4..7 的偏微
         __syncthreads()

Step 3: [COMPUTE with PING]                         ←── 计算 k=8..11 的偏微
         __syncthreads()

Final:   [COMPUTE with PONG?]                       ←── 或取决于实现
```

每次 `__syncthreads()` 确保当前缓冲的数据就绪，然后下一轮计算用当前缓冲、加载下一缓冲。

### 8.3 效果

```
AI(K7) ≈ 1.600 FLOPs/byte  (K1 的 6.60 倍)
```

K7 的计算访存比提升来自两方面：
1. 共享内存 + thread tile + 寄存器缓存（与 K3-K6 相同的复用）
2. 双缓冲重叠了加载延迟，使有效内存带宽利用率接近理论峰值

## 9. 汇总对比与结论

### 9.1 各 Kernel 对比

| Kernel | 全局内存流量 (bytes) | 计算访存比 (FLOPs/byte) | vs K1 | 关键优化手段 |
|--------|---------------------|------------------------|-------|------------|
| K1 Naive | 33,792 | 0.242 | 1.00× | 基准（无优化） |
| K2 SMEM | 9,216 | 0.889 | 3.67× | 共享内存缓存，块内复用 |
| K3 1D Tile | 8,192 | 1.000 | 4.12× | 1D Thread Tile (TM=4) |
| K4 2D Tile | 8,192 | 1.000 | 4.12× | 2D Thread Tile (TM=TN=4) |
| K5 Reg Cache | 8,192 | 1.000 | 4.12× | 显式寄存器缓存 |
| K6 Float4 | 8,192 | 1.000 | 4.12× | float4 向量化加载 |
| K7 DblBuf | 5,120 | 1.600 | 6.60× | 双缓冲延迟隐藏 |

### 9.2 优化路线图

```
       全局内存    共享内存    寄存器      向量化      双缓冲
K1 ──→ K2 ────→ K3 ────→ K4 ──→ K5 ──→ K6 ──→ K7
减少重复    Thread    2D Tile   寄存器    利用总线   重叠
读取       Tile                缓存      带宽      加载/计算
```

### 9.3 从第一性原理总结

SGEMM 优化的本质是：

1. **减少全局内存访问** (K1→K2)：同一数据被多个线程需要时，缓存到更近的存储层次
2. **提高数据复用** (K2→K3→K4)：一个线程一次加载的寄存器数据，用于多次 FMA
3. **利用硬件特性** (K5→K6→K7)：寄存器缓存减少指令延迟、float4 提高总线利用率、双缓冲隐藏加载延迟

这套优化方法的通用性：任何 memory-bound 的 GEMM-like 计算（卷积、Transformer 的 MatMul）都可以用相同的 tile-reuse-pipeline 范式优化。
