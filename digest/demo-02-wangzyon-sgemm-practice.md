## 2. K1: Naive 全局内存实现

### 2.1 核心思想

最直接的方式：每个线程计算 C 的一个元素。线程配置：

- `blockDim = (4, 4)` → 每 block 16 线程
- `gridDim = (4, 4, 1)` → 16 blocks
- 总线程数 = 16 × 16 = 256，恰好覆盖 16×16 的 C

每个线程执行： `C[row][col] = Σ(k=0..15) A[row][k] × B[k][col]`

#### 2.1.5 层次树

```
矩阵 C (16×16, GMEM)
└── Grid: gridDim=(4,4,1), 共 4×4=16 blocks  (行: CEIL_DIV(M,4)=16/4=4, 列: CEIL_DIV(N,4)=16/4=4)
    └── Block: 16 threads (blockDim.x=4, blockDim.y=4)
        └── Thread: 1 element of C
            └── 16 次迭代 K: 每次 2×GMEM读 + 1×FMA
            总 GMEM 流量: (4096+4096+256)×4 = 33,792 bytes
```

### 2.2 访存分析 (GMEM vs SMEM)

**从第一性原理角度**：SGEMM 是典型的 memory-bound 问题。对于 M=N=K=N 的方阵：

- **计算量:** M×N×K = 16×16×16 = 4096 次 FMA（融合乘加）= 8192 FLOPs
- **全局内存 (GMEM) 访问（K1 无缓存复用）:**
  - A 读取 (GMEM): 256 threads × 16 elements = 4096 floats = 16,384 bytes
  - B 读取 (GMEM): 256 threads × 16 elements = 4096 floats = 16,384 bytes
  - C 写入 (GMEM): 256 floats = 1,024 bytes
  - 总 GMEM 流量: 33,792 bytes
- **共享内存 (SMEM) 访问:** 0 bytes（未使用）

```
算术强度 AI (GMEM) = 8192 FLOPs / 33792 bytes ≈ 0.2424 FLOPs/byte
```

**0.2424 意味着什么？** 每从全局内存读取 1 byte 数据，只做 0.2424 次浮点运算。考虑到现代 GPU 的内存带宽（如 A100: ~2 TB/s）和计算能力（~19.5 TFLOPS），要达到计算饱和需要 ~9.75 FLOPs/byte。当前 0.2424 远远不够——**被全局内存带宽严重限制**。

### 2.3 C[0,0] 的完整计算

线程 block(0,0), thread(0,0) 负责 C[0,0]:

```
C[0,0] = A[0,0]×B[0,0] + A[0,1]×B[1,0] + ... + A[0,15]×B[15,0]
```

逐项计算（每一行是一条 FMA）：

| k | A[0,k] | B[k,0] | 乘积 | 累加值 |
|---|--------|--------|------|--------|
| 0 | 5 | 3 | 15 | 15 |
| 1 | 8 | 7 | 56 | 71 |
| 2 | 9 | 0 | 0 | 71 |
| 3 | 5 | 6 | 30 | 101 |
| 4 | 0 | 4 | 0 | 101 |
| 5 | 0 | 8 | 0 | 101 |
| 6 | 1 | 9 | 9 | 110 |
| 7 | 7 | 2 | 14 | 124 |
| 8 | 6 | 2 | 12 | 136 |
| 9 | 9 | 1 | 9 | 145 |
| 10 | 2 | 4 | 8 | 153 |
| 11 | 4 | 6 | 24 | 177 |
| 12 | 5 | 9 | 45 | 222 |
| 13 | 2 | 2 | 4 | 226 |
| 14 | 4 | 0 | 0 | 226 |
| 15 | 2 | 6 | 12 | **238** |

验证: C[0,0] = 238 ✓

### 2.4 K1 的问题

每个线程独立地从全局内存读取所有需要的 A 和 B 元素。**同一元素被重复读取多次**——A[0,0] 被 Block(0,0) 的 16 个线程读取 16 次，还被 Block(0,1)、Block(0,2)、Block(0,3) 的线程读取，总共被每个需要 A[0,k] 来算 C[0,\*] 的线程读取 16×4 = 64 次。

## 3. K2: 共享内存缓存 (SMEM Caching)

### 3.1 核心思想

**第一性原理**: 如果多个线程需要同一数据，在更近的存储层次缓存它。GPU 的共享内存（**共享内存 (SMEM)**）是 block 内所有线程共享的片上 SRAM，延迟 ~20-30 cycles vs 全局内存（**全局内存 (GMEM)**）的 ~300-500 cycles。

**参数:** BM=BN=BK=4, blockDim=(4,4), gridDim=(4,4)

将 K 维分成 4 步（BK=4），每一步：
1. Block 协作从 **全局内存 (GMEM)** 加载 4×4 tile 的 A 和 B 到 **共享内存 (SMEM)**（只从 GMEM 读一次）
2. `__syncthreads()` 确保所有数据就绪
3. 16 个线程从 **共享内存 (SMEM)** 读取数据，计算偏微和

#### 3.1.5 层次树

```
矩阵 C (16×16)
└── Grid: gridDim=(4,4), 共 4×4=16 blocks  (行: 16/BM=16/4=4, 列: 16/BN=16/4=4)
    └── Block: BM=4, BN=4, BK=4 (blockDim=(4,4))
        ├── [GMEM→SMEM] As[4×4] + Bs[4×4] (32 floats = 128 bytes) per K-step
        ├── K 迭代: 16/BK=4 slides along K dimension
        ├── __syncthreads() 确保 SMEM 就绪
        └── Thread: 1 element, 从 SMEM 做 BK=4 次内积
            └── 每个 block 16 线程协作加载 1 次, 共享 SMEM
```

### 3.2 Tile 滑动模式

Block(0,0) 覆盖 C[0:4, 0:4]，K 方向分 4 步滑动：

```
K step 0: A[0:4,0:4] × B[0:4,0:4]   → partial C
K step 1: A[0:4,4:8] × B[4:8,0:4]   → partial C
K step 2: A[0:4,8:12] × B[8:12,0:4]  → partial C
K step 3: A[0:4,12:16] × B[12:16,0:4] → partial C

C[0:4,0:4] = Σ partial
```

### 3.4 访存分析 (GMEM vs SMEM)

**全局内存 (GMEM) 访问 (K2):**

- A 读取 (GMEM): 16 blocks × 16 elements × 4 ksteps = 1024 floats = 4,096 bytes
- B 读取 (GMEM): 16 blocks × 16 elements × 4 ksteps = 1024 floats = 4,096 bytes
- C 写入 (GMEM): 256 floats = 1,024 bytes
- 总 GMEM 流量: **9,216 bytes**

**共享内存 (SMEM) 访问 (K2):**

SMEM 访问分为两类：从 GMEM 写入 SMEM，以及线程从 SMEM 读取做 FMA。

- **SMEM writes (GMEM→SMEM)**: 与 GMEM 读取的 A/B tiles 同量 = 4,096 + 4,096 = **8,192 bytes**
  - 16 blocks × 4 K-steps × (16 floats As + 16 floats Bs) × 4B = 8,192 bytes
- **SMEM reads (线程从 SMEM 读入寄存器做 FMA)**:
  - 每个线程每 K-step: 读 BK=4 个 As 元素 + 读 BK=4 个 Bs 元素 = 8 次 SMEM 读取
  - 每线程总计: 4 K-steps × 8 次 = 32 次 SMEM 读取
  - 256 线程总计: 256 × 32 × 4B = **32,768 bytes**
- **SMEM 总流量 (reads + writes)**: 32,768 + 8,192 = **40,960 bytes**

```
AI (GMEM) = 8192 / 9216 ≈ 0.8889 FLOPs/byte  (K1 的 3.67 倍)
```

**为什么减少了？** 在 K1 中，A[0,0] 被 64 个线程各自从全局内存读取。在 K2 中，A[0,0] 被 block 的所有 16 个线程通过一次全局加载 + 共享内存广播共享。虽然不同 block 之间仍有重复，但 block 内部的重复被消除了。A[0,0] 从被读 64 次降到 4 次（4 个 block 各加载一次）。

## 4. K3: 一维线程 Tile (1D Thread Tile)

### 4.1 核心思想

**第一性原理**: 在 K2 中每个线程只负责 1 个 C 元素，每次 FMA 都要从共享内存读取。如果让每个线程负责多个 C 元素，就可以复用从共享内存加载的寄存器数据——一次加载，多次使用。

**参数:** BM=8, BN=8, BK=4, TM=4. blockDim=(8,2), gridDim=(2,2)

- blockDim.x=8 (N方向线程), blockDim.y=2 (M方向线程)
- 每个 M 方向线程计算 TM=4 个连续 C 元素（沿 M 方向展开）

**线程数:** 2×2×8×2 = 64 线程（vs K2 的 256）
**每线程工作量:** 4 个 C 元素（vs K1/K2 的 1 个）

#### 4.1.5 层次树

```
矩阵 C (16×16)
└── Grid: gridDim=(2,2), 共 2×2=4 blocks  (行: 16/BM=16/8=2, 列: 16/BN=16/8=2)
    └── Block: BM=8, BN=8, BK=4 (blockDim.x=8, blockDim.y=2)
        ├── [GMEM→SMEM] As[8×4] + Bs[4×8] (每步 64 floats, 256 bytes)
        ├── 线程数: 8×2=16  (vs K2的 256 线程)
        └── Thread Tile: TM=4 (沿 M, 1D)
            └── Thread (ty=0,tx=0): 计算 C[0,0]..C[3,0] 共 4 个元素
            └── reg_A[TM][BK] = reg_A[4][4] 从 SMEM 加载一次, 复用 TM=4 次
```

### 4.2 线程映射

Block(0,0) 覆盖 C[0:8, 0:8] = 64 个元素:

| 线程 | 负责的 C 元素 |
|------|-------------|
| (ty=0, tx=0) | C[0,0], C[1,0], C[2,0], C[3,0] |
| (ty=1, tx=0) | C[4,0], C[5,0], C[6,0], C[7,0] |
| (ty=0, tx=1) | C[0,1], C[1,1], C[2,1], C[3,1] |
| (ty=1, tx=1) | C[4,1], C[5,1], C[6,1], C[7,1] |
| ... | ... |
| (ty=0, tx=7) | C[0,7], C[1,7], C[2,7], C[3,7] |
| (ty=1, tx=7) | C[4,7], C[5,7], C[6,7], C[7,7] |

### 4.3 寄存器级操作：thread(ty=0, tx=0)

该线程持有 4 行 A 数据和 1 列 B 数据（对于 kt=0），从 **共享内存 (SMEM)** 读取:

**reg_A (4×4) = A[0:4, 0:4]:**
```
reg_A[0] = [  5,  8,  9,  5 ]
reg_A[1] = [  4,  7,  7,  9 ]
reg_A[2] = [  8,  8,  3,  9 ]
reg_A[3] = [  0,  3,  9,  2 ]
```

**reg_B (4×1) = B[0:4, 0]:**
```
reg_B[0] = 3
reg_B[1] = 7
reg_B[2] = 0
reg_B[3] = 6
```

**FMA 计算（kt=0 偏微）：**

C[0,0] 偏微:
```
reg_A[0][0]×reg_B[0] =  5 ×  3 =  15
reg_A[0][1]×reg_B[1] =  8 ×  7 =  56
reg_A[0][2]×reg_B[2] =  9 ×  0 =   0
reg_A[0][3]×reg_B[3] =  5 ×  6 =  30
                              Σ = 101
```

C[1,0] 偏微:
```

reg_A[1][1]×reg_B[1] =  7 ×  7 =  49
reg_A[1][2]×reg_B[2] =  7 ×  0 =   0
reg_A[1][3]×reg_B[3] =  9 ×  6 =  54
                              Σ = 115
```

C[2,0] 偏微: 8×3 + 8×7 + 3×0 + 9×6 = 24 + 56 + 0 + 54 = **134**

C[3,0] 偏微: 0×3 + 3×7 + 9×0 + 2×6 = 0 + 21 + 0 + 12 = **33**

### 4.4 关键：B 列数据的复用

thread(ty=0, tx=0) 将 B[0:4, 0] 加载一次到寄存器（来自 **共享内存 (SMEM)**），然后与 reg_A 的 4 行分别做点积——**一次加载，4 次使用**。这就是 Thread Tile 复用的本质。

### 4.5 访存分析 (GMEM vs SMEM)

**全局内存 (GMEM) 访问 (K3):**

- A 读取 (GMEM): 4 blocks × 4 ksteps × 32 floats = 512 floats = 2,048 bytes
- B 读取 (GMEM): 4 blocks × 4 ksteps × 32 floats = 512 floats = 2,048 bytes
- C 写入 (GMEM): 1024 floats = 4,096 bytes
- 总 GMEM 流量: **8,192 bytes**

**共享内存 (SMEM) 访问 (K3):**

- 本 kernel 从全局内存读取 2,048 + 2,048 = 4,096 bytes (加载 A/B tiles)
- 本 kernel 从共享内存读取 tiles 供各线程计算

```
AI (GMEM) = 8192 / 8192 ≈ 1.0000 FLOPs/byte  (K1 的 4.12 倍)
```

## 5. K4: 二维线程 Tile (2D Thread Tile)

### 5.1 核心思想

将 thread tile 从 1D 扩展到 2D：一个线程计算一个 TM×TN = 4×4 = **16 个 C 元素**。

**参数:** BM=8, BN=8, BK=4, TM=4, TN=4. blockDim=(2,2), gridDim=(2,2)

- 每个 block 只有 4 个线程！
- 每个线程负责 4×4=16 个 C 元素
- Block(0,0) 的 4 个线程覆盖 C[0:8, 0:8] = 64 元素

#### 5.1.5 层次树

```
矩阵 C (16×16)
└── Grid: gridDim=(2,2), 共 2×2=4 blocks  (行: 16/BM=16/8=2, 列: 16/BN=16/8=2)
    └── Block: BM=8, BN=8, BK=4 (blockDim=(2,2), 仅 4 线程/block!)
        ├── [GMEM→SMEM] As[8×4] + Bs[4×8]
        └── Thread Tile: TM=4, TN=4 (2D, 16 元素/线程)
            └── 4 线程 × 16 元素 = 64 元素 = 1 Block
            └── Thread(0,0): C[0:4,0:4]    Thread(0,1): C[0:4,4:8]
            └── Thread(1,0): C[4:8,0:4]    Thread(1,1): C[4:8,4:8]
```

### 5.2 线程块划分

Block(0,0) 覆盖 C[0:8, 0:8]:

```
+-------------------+-------------------+
| Thread(0,0)       | Thread(0,1)       |
| C[0:4, 0:4]       | C[0:4, 4:8]       |
| 16 elements       | 16 elements       |
+-------------------+-------------------+
| Thread(1,0)       | Thread(1,1)       |
| C[4:8, 0:4]       | C[4:8, 4:8]       |
| 16 elements       | 16 elements       |
+-------------------+-------------------+
```

### 5.3 Thread(0,0) 的寄存器操作

持有 reg_A[4][4]（A[0:4, 0:4]）和 reg_B[4][4]（B[0:4, 0:4]），执行 4×4 矩阵乘法得到 16 个偏微值：

```
[ 101  142  204   41 ]    kt=0 偏微 C[0:4,0:4]
[ 115  127  211   59 ]
[ 134  150  222   61 ]
[  33   61  102   16 ]
```

每列是一个独立的内积：reg_A 的行向量与 reg_B 的第 j 列做点积。

### 5.4 寄存器复用倍数

在 K4 中，reg_B[0][0]（=B[0,0]=3）被用于计算 C[0,0]、C[1,0]、C[2,0]、C[3,0]——**一次加载，4 次使用**。同理 reg_A[0][0] 被用于计算 C[0,0]、C[0,1]、C[0,2]、C[0,3]——也是 **1:4 复用**。

总复用率：每个从共享内存加载的 float 平均用于 **4 次 FMA**（2D tile 下是 TM×TN/BK = 16/4 = 4 倍）。

### 5.5 访存分析 (GMEM vs SMEM)

K4 与 K3 相同的 GMEM 数据流（相同 SMEM tile 尺寸），所以 AI 相同：

- GMEM 总流量: **8,192 bytes**
- SMEM 总流量: 与 K3 相同量级

```
AI (GMEM) ≈ 1.0000 FLOPs/byte
```

但线程数减少（64→16），每个线程的寄存器压力增大，ILP（指令级并行）潜力增加。

## 6. K5: 寄存器缓存 (Register Caching)

### 6.1 核心思想

**第一性原理**: 在 K4 中，每次 FMA 的 operands 来自 **共享内存 (SMEM)**。共享内存虽快（~20-30 cycles），但寄存器更快（~0 cycles 延迟，如果编译器优化得当）。

K5 将每个线程需要的 As/Bs 片段显式加载到局部寄存器数组：

```cuda
float reg_A[TM][BK];  // reg_A[4][4] = 16 registers
float reg_B[BK][TN];  // reg_B[4][4] = 16 registers
```

然后对 reg_A 和 reg_B 执行 4×4×4 的 FMA 循环——所有操作数都在寄存器中。

#### 6.1.5 层次树

```
矩阵 C (16×16)
└── Grid: gridDim=(2,2), 共 2×2=4 blocks  (行: 16/BM=16/8=2, 列: 16/BN=16/8=2)
    └── Block: BM=8, BN=8, BK=4 (blockDim=(2,2), 仅 4 线程/block!)
        ├── [GMEM→SMEM] As[8×4] + Bs[4×8]
        ├── [SMEM→RF] 显式寄存器缓存: reg_A[TM][BK] + reg_B[BK][TN] (32 regs/thread)
        └── Thread Tile: TM=4, TN=4 (2D, 16 元素/线程)
            ├── [SMEM→RF] reg_A[4][4] ← As[ty*4:ty*4+4, :]  (加载到寄存器)
            ├── [SMEM→RF] reg_B[4][4] ← Bs[:, tx*4:tx*4+4]
            └── [RF: FMA] reg_C[tm][tn] += reg_A[tm][k] * reg_B[k][tn]
                全部操作数在寄存器中, 无 SMEM 重复访问
```

### 6.2 内存层次

```
全局内存 (GMEM)           ~300-500 cycles, ~1-2 TB/s
    ↓ 协作加载
共享内存 (SMEM)            ~20-30 cycles, ~10 TB/s (per SM)
    ↓ 寄存器加载
寄存器 (Register File)     ~0 cycles (编译器调度)
    ↓ FMA 指令
计算单元 (CUDA Core)       1 cycle per FMA (warp)
```

**关键洞察**: 数据从全局内存到共享内存，再加载到寄存器。一旦数据在寄存器中，后续 FMA 完全不受内存延迟影响——这是 "register tiling" 的本质。

### 6.3 K5 vs K4 的区别

| 方面 | K4 | K5 |
|------|----|----|
| GMEM 加载 | 协作加载 As/Bs tiles | 不变 |
| SMEM 加载 | 隐式（编译器调度） | 不变 |
| FMA 操作数来源 | 共享内存 (SMEM) | 寄存器 |
| 编译器可优化性 | 中等 | 高（显式展开） |
| 计算访存比 AI (GMEM) | 1.0000 | 1.0000（GMEM 数据量相同） |

K5 的改进不在访存比，而在**指令执行效率**——寄存器操作数让编译器更容易做指令重排和延迟隐藏。

## 7. K6: Float4 向量化加载

### 7.1 核心思想

CUDA 的内存总线宽度是 128-bit（32 bytes × 4 = 128 bits）。用 `float`（32-bit）加载每次只用总线容量的 1/4。`float4` 一次加载 128 bits = 4 个 float，充分利用总线。

```cuda
// K5: 4 次 32-bit 加载 (GMEM → SMEM)
smem[0] = global[0];
smem[1] = global[1];
smem[2] = global[2];
smem[3] = global[3];

// K6: 1 次 128-bit 加载 (GMEM → SMEM)
reinterpret_cast<float4*>(smem)[0] = reinterpret_cast<float4*>(global)[0];
```

#### 7.1.5 层次树

```
矩阵 C (16×16)
└── Grid: gridDim=(2,2), 共 2×2=4 blocks  (行: 16/BM=16/8=2, 列: 16/BN=16/8=2)
    └── Block: BM=8, BN=8, BK=4 (blockDim=(2,2), 仅 4 线程/block!)
        ├── [GMEM→SMEM] float4 加载: 8 次 128-bit 事务 (vs K5 的 32 次 32-bit)
        │   每行 4 个 float 打包: float4(A[i,0], A[i,1], A[i,2], A[i,3])
        │   A tile (8×4): 8行×1float4 = 8 事务 | B tile (4×8): 8列×1float4 = 8 事务
        ├── [SMEM→RF] 显式寄存器缓存 reg_A[TM][BK] + reg_B[BK][TN]
        └── Thread Tile: TM=4, TN=4 (2D, 16 元素/线程)
```

### 7.2 具体示例

Block(0,0), kt=0 时加载 A[0:8, 0:4] 从 **全局内存 (GMEM)** 到 **共享内存 (SMEM)**。

用 float4 加载，每行 4 个 float 打包为一次加载：

```
float4(A[0,0], A[0,1], A[0,2], A[0,3])
  = float4(5, 8, 9, 5)

float4(A[1,0], A[1,1], A[1,2], A[1,3])
  = float4(4, 7, 7, 9)
```

8 行 × 每行 1 次 float4 = **8 次事务**（vs K5 的 32 次）
4 列 B × 每次 1 次 float4 = **8 次事务**（vs K5 的 32 次）

### 7.3 访存分析

- 等效 float 读取 (GMEM): A=512, B=512（与 K5 相同）
- GMEM 事务数减少 4 倍
- GMEM 总流量: **8,192 bytes**（与 K5 相同）
- AI (GMEM): **1.0000 FLOPs/byte**

### 7.4 效果

数据量相同，内存事务数减少 4 倍。对于 bandwidth-bound 的 SGEMM，更高的总线利用率直接转化为更短的加载时间。

## 8. K7: 双缓冲 (Double Buffering)

### 8.1 核心思想

**第一性原理**: Pipeline 化。**全局内存 (GMEM)** 加载和计算是两个独立的硬件单元（内存控制器 vs CUDA Cores）。如果让它们串行执行，总时间 = 加载时间 + 计算时间。如果让它们重叠，总时间 ≈ max(加载时间, 计算时间)。

双缓冲使用两套 **共享内存 (SMEM)**（PING 和 PONG），交替用于加载和计算。

#### 8.1.5 层次树

```
矩阵 C (16×16)
└── Grid: gridDim=(2,2), 共 2×2=4 blocks  (行: 16/BM=16/8=2, 列: 16/BN=16/8=2)
    └── Block: BM=8, BN=8, BK=4 (blockDim=(2,2), 仅 4 线程/block!)
        ├── SMEM 双缓冲: PING[8×4+4×8] + PONG[8×4+4×8] (双倍 SMEM)
        ├── [GMEM→SMEM] float4 加载到当前写入缓冲
        ├── [SMEM→RF] 显式寄存器缓存 reg_A[TM][BK] + reg_B[BK][TN]
        ├── Thread Tile: TM=4, TN=4 (2D, 16 元素/线程)
        └── Pipeline 时间线 (Block(0,0)):
            ├── k_step=0: [LOAD→PING]                          # 预加载
            ├── k_step=1: [LOAD→PONG] ∥ [COMPUTE with PING]    # 加载与计算重叠
            ├── k_step=2: [LOAD→PING]  ∥ [COMPUTE with PONG]
            ├── k_step=3:               [COMPUTE with PING]     # 最后一轮
            └── __syncthreads() 只在每轮结束时同步一次
```

### 8.2 Ping-Pong 时间线

Block(0,0) 在 4 个 K 步上的执行：

```
Step 0: [LOAD  A[0:8,0:4], B[0:4,0:8]  GMEM→PING]    (预加载，无计算)
         __syncthreads()

Step 1: [LOAD  A[0:8,4:8], B[4:8,0:8]  GMEM→PONG]    ←── 加载与计算重叠
         [COMPUTE with PING (SMEM)]                   ←── 计算 k=0..3 的偏微
         __syncthreads()

Step 2: [LOAD  A[0:8,8:12], B[8:12,0:8] GMEM→PING]   ←── 加载与计算重叠
         [COMPUTE with PONG (SMEM)]                   ←── 计算 k=4..7 的偏微
         __syncthreads()

Step 3: [COMPUTE with PING (SMEM)]                    ←── 计算 k=8..11 的偏微
         __syncthreads()
```

每次 `__syncthreads()` 确保当前缓冲的数据就绪，然后下一轮计算用当前缓冲、加载下一缓冲。

### 8.3 访存分析 (GMEM vs SMEM)

**全局内存 (GMEM) 访问 (K7):**

- A 读取 (GMEM): 128 float4 事务 = 512 floats = 2,048 bytes
- B 读取 (GMEM): 128 float4 事务 = 512 floats = 2,048 bytes
- C 写入 (GMEM): 256 floats = 1,024 bytes
- 总 GMEM 流量: **5,120 bytes**

```
AI (GMEM) = 8192 / 5120 ≈ 1.6000 FLOPs/byte  (K1 的 6.60 倍)
```

K7 的计算访存比提升来自两方面：
1. 共享内存 + thread tile + 寄存器缓存（与 K3-K6 相同的复用）
2. 双缓冲重叠了加载延迟，使有效内存带宽利用率接近理论峰值

## 9. 汇总对比与结论

### 9.1 各 Kernel 对比

| Kernel | GMEM bytes | SMEM bytes | AI (GMEM) | AI (SMEM) | vs K1 | 关键优化手段 |
|--------|-----------|-----------|-----------|-----------|-------|------------|
| K1 Naive | 33,792 | 0 | 0.2424 | - | 1.00× | 基准（无优化） |
| K2 SMEM | 9,216 | ~40,960 | 0.8889 | - | 3.67× | 共享内存 (SMEM) 缓存，块内复用 |
| K3 1D Tile | 8,192 | ~20,480 | 1.0000 | - | 4.12× | 1D Thread Tile (TM=4) |
| K4 2D Tile | 8,192 | ~20,480 | 1.0000 | - | 4.12× | 2D Thread Tile (TM=TN=4) |
| K5 Reg Cache | 8,192 | ~20,480 | 1.0000 | - | 4.12× | 显式寄存器缓存 |
| K6 Float4 | 8,192 | ~20,480 | 1.0000 | - | 4.12× | float4 向量化加载 |
| K7 DblBuf | 5,120 | ~20,480 | 1.6000 | - | 6.60× | 双缓冲延迟隐藏 |

> **注:** SMEM bytes 列表示从共享内存的总访问量（读取 + 写入）。K3-K7 的 GMEM bytes 相同（8,192），因为它们使用相同的 tile 尺寸从全局内存加载数据，优化手段针对的是 SMEM 访问模式、寄存器缓存和加载效率，而非进一步减少 GMEM 流量。K7 的 GMEM 减少源于双缓冲减少了重复加载。

### 9.2 访存路径总结

每个 kernel 的数据流向：

| Kernel | GMEM→SMEM | SMEM→Register | 关键特点 |
|--------|-----------|---------------|---------|
| K1 | 无 SMEM | 无 | 每次 FMA 直接从 GMEM 读取 |
| K2 | 加载 tiles | 每 FMA 从 SMEM 读取 | 块内复用消除 GMEM 重复 |
| K3 | 同 K2 | 1D tile: B列复用4次 | 每线程算4个C元素 |
| K4 | 同 K3 | 2D tile: A行和B列各复用4次 | 每线程算16个C元素 |
| K5 | 同 K4 | 显式 reg_A/reg_B 缓存 | 编译器可做指令重排 |
| K6 | float4 加载 | 同 K5 | 128-bit 总线利用率 4× |
| K7 | ping-pong 缓冲 | 同 K5/K6 | 加载/计算重叠 |

### 9.3 优化路线图

```
       全局内存    共享内存    寄存器      向量化      双缓冲
K1 ──→ K2 ────→ K3 ────→ K4 ──→ K5 ──→ K6 ──→ K7
减少 GMEM    Thread    2D Tile   寄存器    利用总线   重叠
重复读取    Tile                缓存      带宽      加载/计算
```

### 9.4 从第一性原理总结

SGEMM 优化的本质是：

1. **减少全局内存 (GMEM) 访问** (K1→K2)：同一数据被多个线程需要时，缓存到更近的 **共享内存 (SMEM)**
2. **提高数据复用** (K2→K3→K4)：一个线程一次加载到寄存器的数据（来自 SMEM），用于多次 FMA
3. **利用硬件特性** (K5→K6→K7)：寄存器缓存减少指令延迟、float4 提高总线利用率、双缓冲隐藏加载延迟

这套优化方法的通用性：任何 memory-bound 的 GEMM-like 计算（卷积、Transformer 的 MatMul）都可以用相同的 tile-reuse-pipeline 范式优化。
