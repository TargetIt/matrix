## 2. Kernel 1：Naive 实现

### 2.1 核心思想

CUDA 编程模型中，计算被组织为三层层次结构：

```
Grid (网格)
├── Block (0,0) ── 包含最多 1024 个线程
├── Block (0,1)
├── ...
└── Block (gridDim.x-1, gridDim.y-1)
    └── Thread (threadIdx.x, threadIdx.y)
```

在 Naive Kernel 中，每个线程负责计算 C 的一个元素。对于 M=N=K=16，使用 `blockDim=(4,4), grid=(4,4)`，共 256 个线程，每个计算 C 矩阵的一个元素。

线程 (threadIdx.x, threadIdx.y) 在 block (blockIdx.x, blockIdx.y) 中计算：

```
i = blockIdx.x * 4 + threadIdx.x   // 全局行索引
j = blockIdx.y * 4 + threadIdx.y   // 全局列索引
C[i,j] = Σ(k=0..15) A[i,k] * B[k,j]
```

### 2.2 从第一性原理分析

**为什么 Naive 实现很慢？** 每个线程需要从全局内存（GMEM）加载 A 的一整行（16个float = 64B）和 B 的一整列（16个float = 64B）。总共 256 个线程 × 128B = 32KB 的读取量。但是相邻线程加载的 A 行不是连续的——这导致无法利用 GPU 的**合并内存访问（Coalescing）**特性，实际产生了大量浪费的带宽。在 A6000 上这个 Kernel 仅达到 300 GFLOPs（理论的 1.3%）。

### 2.4 内存访问模式的问题

```
Naive 的 A 矩阵访问模式（非合并）:

  Thread (0,0): A[0,0] A[0,1] A[0,2] ... A[0,15]
  Thread (0,1): A[0,0] A[0,1] A[0,2] ... A[0,15]  ← 和 (0,0) 相同！
  Thread (1,0): A[1,0] A[1,1] A[1,2] ... A[1,15]
  ...

同一 warp 中的线程访问的是 A 的不同行（非连续地址），无法合并。
```

## 3. Kernel 2：全局内存合并访问

### 3.1 核心思想

**Warp** 是 GPU 执行的最小单元，包含 32 个线程。当同一个 warp 中的线程访问**连续的内存地址**时，GPU 可以将这些访问合并为一次大事务（如 128B），大幅减少内存事务次数。

```
合并访问示例:
  Thread 0: addr 0x1000  ┐
  Thread 1: addr 0x1004  ├─→ 合并为一次 128B 读取
  Thread 2: addr 0x1008  │
  ...                    │
  Thread 31: addr 0x107C ┘
```

### 3.2 从第一性原理分析

**为什么合并访问有效？** GPU 的内存控制器同一时间可以处理 32B、64B 或 128B 的事务。32 个线程各读取 4B 的 float，地址连续时 = 128B，刚好一次事务完成。如果地址不连续，则需要 32 次独立事务 → 带宽利用率暴跌到 ~5%。

### 3.3 Kernel 2 的实现策略

将 `blockDim` 从 `(4,4)` 改为 `(16)`，即一维线程布局，但保持每 block 256 个线程（16×16 块的 256 个线程里实际上只用 16 个）。通过修改线程索引映射，使得连续 `threadIdx.x` 的线程访问连续的内存地址：

```
// Kernel 2: 一维线程索引 + 合并访问
i = blockIdx.x * BLOCKSIZE + (threadIdx.x / BLOCKSIZE)
j = blockIdx.y * BLOCKSIZE + (threadIdx.x % BLOCKSIZE)
```

在我们的 demo 中：
- `blockDim = (16)`, `grid = (16, 16)` （共 256 blocks，每 block 16 线程）
- 但实际上一个 block 的 16 个线程分别负责 C 中同一行不同列的计算
- 连续 threadIdx.x 的线程访问 A 的**同一行**但不同元素 → 地址连续 → 合并访问

### 3.4 效果

优化后内存吞吐从 15GB/s 提升到 110GB/s，GFLOPs 从 300 提升到 ~2000（8.5% of cuBLAS）。在我们的 16×16 demo 中，计算结果与 Naive 完全一致，因为数学上等价。

## 4. Kernel 3：共享内存缓存分块

### 4.1 核心思想

GPU 上除了大容量的全局内存（GMEM, ~80GB），还有片上共享内存（SMEM, ~48KB/block）。SMEM 的带宽可达 ~12TB/s，比 GMEM 的 ~750GB/s 高一个数量级。

**策略**：将 A 和 B 的小块加载到 SMEM 中，然后在 SMEM 上执行尽可能多的计算，减少 GMEM 访问。

### 4.2 从第一性原理分析

**为什么 SMEM 缓存有效？** 矩阵乘法有大量的数据复用——A 的每一行被 N 个输出元素复用，B 的每一列被 M 个输出元素复用。通过将数据缓存到 SMEM，block 内的所有线程可以共享同一份数据，避免重复从 GMEM 加载。

**关键公式**：对每个 K-tile（大小 BK），block 内的线程协同将 `BM×BK` 的 A-tile 和 `BK×BN` 的 B-tile 加载到 SMEM，然后所有线程在 SMEM 上执行内积。

### 4.3 具体计算演示 (BM=BN=BK=4)

对于 Block (0,0)，第一轮 K-tile (bk_start=0)，加载到 SMEM 的内容：

**As (4×4 tile of A, 行 0-3, 列 0-3):**

```
  [  0.496714   -0.138264    0.647689    1.523030 ]
  [ -1.012831    0.314247   -0.908024   -1.412304 ]
  [ -0.013497   -1.057711    0.822545   -1.220844 ]
  [  0.343618   -1.763040    0.324084   -0.385082 ]
```

**Bs (4×4 tile of B, 行 0-3, 列 0-3):**

```
  [  1.266911   -0.707669    0.443819    0.774634 ]
  [  1.163164    0.010233   -0.981509    0.462103 ]
  [  0.280992   -0.622699   -0.208122   -0.493001 ]
  [ -0.020902    0.117327    1.277665   -0.591571 ]
```

**线程 (0,0) 的第一轮部分内积（第一部分 dotIdx=0..3）：**

```
Part1[0,0] = As[0,0]*Bs[0,0] + As[0,1]*Bs[1,0] + As[0,2]*Bs[2,0] + As[0,3]*Bs[3,0]

  0.496714 ×  1.266911 =  0.629293
 -0.138264 ×  1.163164 = -0.160824
  0.647689 ×  0.280992 =  0.181995
  1.523030 × -0.020902 = -0.031834
──────────────────────────────
Part1[0,0] = 0.618630
```

经过 4 轮 K-tile (bk_start = 0, 4, 8, 12)，累加所有部分内积，最终得到：

```
Part1[0,0] = 0.618630   (bk_start=0, 列 0-3)
Part2[0,0] = ...        (bk_start=4, 列 4-7)
Part3[0,0] = ...        (bk_start=8, 列 8-11)
Part4[0,0] = ...        (bk_start=12, 列 12-15)
────────────────────────
C[0,0]     = 2.011001 ✓
```

**Tile 遍历的可视化:**

```
矩阵 A (16×16):              矩阵 B (16×16):
┌────────┬────────┬────────┬────────┐   ┌────────┬───┬───┬───┐
│ Tile 0 │ Tile 1 │ Tile 2 │ Tile 3 │   │ Tile 0 │   │   │   │

├────────┼────────┼────────┼────────┤   │ Tile 1 │   │   │   │
│   ..   │   ..   │   ..   │   ..   │   ├────────┼───┼───┼───┤
│        │        │        │        │   │ Tile 2 │   │   │   │
│        │        │        │        │   ├────────┼───┼───┼───┤
│        │        │        │        │   │ Tile 3 │   │   │   │
└────────┴────────┴────────┴────────┘   └────────┴───┴───┴───┘

每次迭代: 加载 A 的一个 4×4 Tile 和 B 的一个 4×4 Tile 到 SMEM
然后所有线程在 SMEM 上做内积，结果累加到寄存器中的 partial sum
```

### 4.4 性能分析

这个 Kernel 达到 ~2980 GFLOPs（12.8% of cuBLAS），比上一版提升约 50%。提升没有预期的大，因为 L1 Cache 已经在上一版 Kernel 中提供了较好的命中率。`__syncthreads()` 的同步开销也限制了一部分性能。

## 5. Kernel 4：1D Blocktiling

### 5.1 核心思想

Kernel 3 中每个线程仍只计算 C 的 **1 个元素**。这意味着每做一次 FMA（乘加），就需要从 SMEM 加载 2 个 float。即算术强度（Arithmetic Intensity, AI）非常低。

**关键洞察**：让每个线程计算 C 的**多个元素**（一行中的多个），这样每个线程可以在寄存器中保存中间结果，对 SMEM 的加载可以跨多个输出复用。

```
Kernel 3 (每个线程 1 个结果):    Kernel 4 (每个线程 TM=4 个结果):
  Thread(0,0):                  Thread(0,0):
    Load As[row0], Bs[col0]       Load As[row0], Bs[col0]
    FMA → C[0,0]                  FMA → C[0,0]
    (下一个线程)                   Load As[row1], Bs[col0]
                                   FMA → C[1,0]
                                   ...
                                   (TM=4 个结果在一个线程中计算)
```

### 5.2 从第一性原理分析

**为什么每个线程计算多个结果更快？** 内核瓶颈在于 SMEM 带宽。每个 FMA 需要 2 个 SMEM 读取。通过让每个线程做更多的计算（TM 个结果），可以复用 B 的值（`Btmp` 缓存技术），将每个结果的 SMEM 访问次数从 `2*BK` 降低到 `BK*(1 + 1/TM)`。当 TM=1 时为 2×，TM=8 时仅为 1.125×——减少了 44% 的 SMEM 访问。

### 5.3 参数配置

| 参数 | 值 |
|------|-----|
| BM | 8 (block 覆盖 C 的行数) |
| BN | 8 (block 覆盖 C 的列数) |
| BK | 4 (K 维度分块大小) |
| TM | 4 (每个线程计算的行数) |

对于 M=N=K=16：
- 2×2 = 4 个 blocks
- 每个 block 有 `(BM/TM) * BN = (8/4) * 8 = 16` 个线程

### 5.4 具体计算演示

**Block (0,0), Thread (threadRow=0, threadCol=0):**

该线程负责计算 C 的一个 1D 条带：`C[0,0], C[1,0], C[2,0], C[3,0]`

伪代码：
```
threadResults = [0, 0, 0, 0]  // TM=4

for bk_start in [0, 4, 8, 12]:      // 4 个 K-tile
    加载 As(8×4), Bs(4×8) 到 SMEM
    
    for dotIdx in 0..3:               // BK=4
        Btmp = Bs[dotIdx, threadCol]  // 缓存 B 的值
        for resIdx in 0..3:           // TM=4
            threadResults[resIdx] += As[resRow+resIdx, dotIdx] * Btmp
```

预期结果（匹配 Ground Truth）：
```
C[0,0] =  2.011001 ✓
C[1,0] = -4.049048 ✓
C[2,0] = -1.601734 ✓
C[3,0] =  1.918840 ✓
```

## 6. Kernel 5：2D Blocktiling — 提高算术强度

### 6.1 核心思想

Kernel 4 中每个线程计算 **1D 条带**（TM×1 个 C 元素）。进一步优化：让每个线程计算 **2D 小块**（TM×TN 个 C 元素），进一步提高数据复用率。

```
Kernel 4 (1D: TM×1):           Kernel 5 (2D: TM×TN):
  线程计算 C 的一列              线程计算 C 的一个子块
  
  ████                          ████
  ████  ← 一个线程              ████  ← 一个线程计算 4×4=16 个元素
  ████                          ████
  ████                          ████
```

### 6.2 从第一性原理分析

**为什么 2D 比 1D 更好？** 2D 计算模式允许同时复用 A 和 B 的数据：

- 1D (TM×1)：只能复用 B 的一个元素
- 2D (TM×TN)：可以同时复用 A 的一个元素（在 TN 个输出中）和 B 的一个元素（在 TM 个输出中）

这进一步减少了每个结果的平均 SMEM 访问次数。

### 6.3 参数配置 (demo)

| 参数 | 值 |
|------|-----|
| BM | 8 |
| BN | 8 |
| BK | 4 |
| TM | 4 |
| TN | 4 |

每个线程计算 TM×TN = 4×4 = 16 个 C 元素。

对于 M=16, N=16：
- 2×2 = 4 个 blocks
- 每个 block 有 `(BM/TM) * (BN/TN) = (8/4)*(8/4) = 4` 个线程

### 6.4 具体计算演示

**Block (0,0), Thread (thr_row=0, thr_col=0):**

该线程负责计算 C 的 **4×4 子块**：`C[0:4, 0:4]`

```
线程的子块:
  ┌─────────────────────────────────────────┐
  │ C[0,0]    C[0,1]    C[0,2]    C[0,3]   │
  │  2.011001  -0.349316  2.920266 -4.171154│
  │                                         │
  │ C[1,0]    C[1,1]    C[1,2]    C[1,3]   │
  │ -4.049048   4.066224 -5.477256 -1.483895│
  │                                         │
  │ C[2,0]    C[2,1]    C[2,2]    C[2,3]   │
  │ -1.601734   1.900657 -2.519432  0.639217│
  │                                         │
  │ C[3,0]    C[3,1]    C[3,2]    C[3,3]   │
  │  1.918840   0.638398  6.441704 -3.475225│
  └─────────────────────────────────────────┘
```

伪代码：
```
threadResults = 4×4 零矩阵

for bk_start in [0, 4, 8, 12]:
    加载 As(8×4), Bs(4×8) 到 SMEM
    
    for dotIdx in 0..3:                // BK=4
        for resRow in 0..3:            // TM=4
            for resCol in 0..3:        // TN=4
                threadResults[resRow,resCol] += As[resRow, dotIdx] * Bs[dotIdx, resCol]
```

所有 16 个元素均与 Ground Truth 一致 ✓

## 7. Kernel 6：向量化内存访问

### 7.1 核心思想

GPU 的 SMEM 支持 128-bit（即 `float4`）的加载/存储指令。这意味着可以用一条指令加载 4 个连续的 float，而不是 4 条独立指令。

```
非向量化:                       向量化:
  ld.shared.f32 r1, [addr]        ld.shared.v4.f32 {r1,r2,r3,r4}, [addr]
  ld.shared.f32 r2, [addr+4]
  ld.shared.f32 r3, [addr+8]
  ld.shared.f32 r4, [addr+12]
```

### 7.2 从第一性原理分析

**为什么向量化有效？** SMEM 有多个 bank（32 个），bank 冲突会序列化访问。使用 128-bit 加载可以减少指令数（指令发射压力），同时编译器可以更好地调度指令隐藏延迟。在我们的 demo 中（BK=4），As 的每一行自动就是 4 个连续的 float——天然适合 `float4` 加载。

### 7.3 在 16×16 demo 中的体现

对于 BK=4，As 的每行恰好是 4 个 float：
```
As[0] = {0.496714, -0.138264, 0.647689, 1.523030}  ← 一次 float4 加载
```
这与 Kernel 5 计算结果完全一致。原始文章中这个优化将性能从 68.7% 提升到 78.4% of cuBLAS。

## 8. Kernel 10：Warp 级分块

### 8.1 核心思想

一个 Block 内的线程进一步按 Warp 组织。Warp（32 线程）可以协作加载数据到寄存器，进一步减少 SMEM 访问。

**策略**：
1. 将 Block 的工作按 Warp 划分
2. 每个 Warp 负责 C 的一个子区域
3. Warp 内的线程协作加载数据，最大化数据复用

### 8.2 Demo 配置

| 参数 | 值 | 含义 |
|------|-----|------|
| BM | 8 | Block 覆盖的行 |
| BN | 8 | Block 覆盖的列 |
| BK | 8 | K-tile 大小 |
| WM | 4 | Warp 协作的行数 |
| WN | 4 | Warp 协作的列数 |
| WMIter | 2 | Warp 内沿 M 的迭代次数 |
| WNIter | 2 | Warp 内沿 N 的迭代次数 |
| TM | 4 | 每线程计算的行数 |
| TN | 4 | 每线程计算的列数 |

Warp 区域 = `WMIter * TM × WNIter * TN = 8 × 8` = 整个 Block

### 8.3 Warp 工作划分 (Block 0,0)

```
Block (0,0) 覆盖 C[0:8, 0:8]:
┌───────────────────────┬───────────────────────┐
│ Thread (0,0)          │ Thread (0,1)          │
│ C[0:4, 0:4]           │ C[0:4, 4:8]           │
│                       │                       │
│  2.011 -0.349  2.920  │ -4.171  0.633  3.845  │
│ -4.049  4.066 -5.477  │ -1.484 -2.004 -1.733  │
│ -1.602  1.901 -2.519  │  0.639  3.423  3.659  │
│  1.919  0.638  6.442  │ -3.475  2.352  1.639  │
├───────────────────────┼───────────────────────┤
│ Thread (1,0)          │ Thread (1,1)          │
│ C[4:8, 0:4]           │ C[4:8, 4:8]           │
│                       │                       │
│ -1.120 -1.615 -0.660  │  4.736 -4.136 -8.799  │
│  3.155  0.246  2.173  │ -2.456  1.656 -0.873  │
│  3.029  1.389  6.023  │ -2.602  5.306  7.089  │
│  8.466 -0.502  0.786  │  5.150  3.032  1.179  │
└───────────────────────┴───────────────────────┘
```

4 个"逻辑线程"（每 warp 内 WMIter×WNIter=4），每个计算 4×4=16 个 C 元素。

### 8.4 从第一性原理分析

**为什么 Warp tiling 有效？** 在较新的 GPU 架构（Volta+）上，warp 内的线程可以通过 `__shfl_sync` 等 warp 级原语直接在寄存器间交换数据，完全绕过 SMEM。这可以将每个 FMA 的 SMEM 访问次数降到接近 0——数据只需加载一次到 warp 的寄存器文件中，然后在 warp 内广播。这是接近 cuBLAS 性能的关键技术。原始文章达到 21.8 TFLOPs（93.7% of cuBLAS）。

## 9. 性能分析：Occupancy、算术强度与 Roofline 模型

### 9.1 Occupancy

**定义**：每个 SM 上的活跃 warp 数 / 最大可能活跃 warp 数。

对 Kernel 3（BM=BN=BK=32 在 A6000 上）:
```
每个 Block: 1024 threads = 32 warps, 37 registers/thread, 8KB SMEM
每个 SM:    max 1536 threads = 48 warps, 100KB SMEM

SMEM 限制:  100KB / 8KB = 12.5 → 最多 12 blocks
Thread 限制: 1536 / 1024 = 1.5 → 最多 1 block  ← 瓶颈！
Register 限制: 65536 / (1280*32) ≈ 1.6 → 最多 1 block

Occupancy = 32 active warps / 48 max warps = 66.7%
```

**关键洞察**：高 Occupancy 不总是好事。当算术强度很高时，低 Occupancy 足以隐藏延迟（因为有足够多的独立指令可发射）。这在 Volkov 的博士论文中被总结为"Cusp Behavior"。

### 9.2 算术强度 (Arithmetic Intensity)

**定义**：AI = FLOPs / Bytes transferred

```
总 FLOPs = 2 × M × N × K = 2 × 16³ = 8,192

各 Kernel 的内存传输 (demo 16×16):

K1 Naive:    每个线程加载 2×16 floats × 256 threads = ~128KB (最坏情况, 无缓存)
K3 SMEM:     BM×BK + BK×BN = 4×4+4×4 = 32 floats/K-tile/block
             × 4 K-tiles × 16 blocks × 4B = 8,192B GMEM reads
             AI ≈ 8192 / 8192 ≈ 1.0 FLOPs/Byte
             
K5 2D:       同样 ~8KB GMEM reads，但更多 SMEM→Register 复用
             AI ≈ 2.0 FLOPs/Byte (因 SMEM 复用效率更高)
             
K10 Warp:    AI ≈ 8.0 FLOPs/Byte (因 warp 级寄存器复用)
```

### 9.3 Roofline 模型

```
峰值 fp32 算力 (A6000):  ~30 TFLOPs/s
峰值 GMEM 带宽 (A6000):  ~768 GB/s
Ridge Point (算力/带宽):  30,000 / 768 ≈ 39 FLOPs/Byte

┌──────────────────────────────────────────────┐
│                     Roofline                 │
│  TFLOPs/s                                    │
│  30 ┤                              ╭─────────│ ← 算力上限 (Compute Bound)
│     │                           ╭──╯         │
│  20 ┤                     ╭─────╯  K10       │
│     │              ╭──────╯   K6             │
│  10 ┤       ╭──────╯  K5                    │
│     │  ╭────╯ K4                             │
│   1 ┤──╯ K3                                  │
│     ├────┼────┼────┼────┼────┼────┼──────    │
│     0.1   1   10  39  100              AI    │
│     ← Memory Bound →│← Compute Bound →       │
└──────────────────────────────────────────────┘

K1/K2:    AI < 0.1,  深度 Memory Bound
K3:       AI ≈ 0.5,  Memory Bound  
K4-K6:    AI ≈ 1-3,  过渡区
K10:      AI ≈ 8,    接近 Compute Bound
cuBLAS:   AI ≈ 245,  深度 Compute Bound (利用了 Tensor Cores)
```

**关键结论**：所有优化都在做同一件事——用更少的字节传输完成相同的 FLOPs。当 AI 超过 Ridge Point（39 FLOPs/Byte），kernel 变成 Compute Bound，进一步优化应聚焦于提高指令级并行度和利用 Tensor Cores。

## 10. 总结

| Kernel | 优化技术 | 关键原理 | 相对 cuBLAS |
|--------|---------|---------|------------|
| 1 Naive | 无 | 基础矩阵乘法 | 1.3% |
| 2 Coalescing | 合并 GMEM 访问 | 连续地址 → 减少内存事务 | 8.5% |
| 3 SMEM | 共享内存缓存 | 片上缓存减少 GMEM 访问 | 12.8% |
| 4 1D Blocktiling | 每线程计算多个输出 | 提高算术强度，复用 B | 36.5% |
| 5 2D Blocktiling | 每线程计算 2D 子块 | 同时复用 A 和 B | 68.7% |
| 6 Vectorized | float4 SMEM 加载 | 减少指令数，提高吞吐 | 78.4% |
| 10 Warptiling | Warp 级协作 + 寄存器复用 | 减少 SMEM 压力 | 93.7% |

**第一性原理总结**：

1. **内存层次结构是关键**：GMEM (慢) → L1/L2 Cache → SMEM (快) → Register (最快)。每次将数据"推"到更靠近计算单元的地方，性能就提升一个台阶。

2. **算术强度是统一的优化目标**：所有技术本质上都在提高 AI = FLOPs / Bytes。Naive Kernel 每个 FMA 需要 ~128B 的内存传输（AI≈0.008），而 cuBLAS 的 AI ≈ 245 FLOPs/Byte。

3. **GPU 是吞吐量架构，不是延迟架构**：单个指令延迟（FMA=4 cycles）不是瓶颈，关键是能否持续"喂饱"计算单元。高 Occupancy 帮助隐藏延迟，但高 AI 时不需要高 Occupancy。

4. **渐进式优化是理解系统的唯一方法**：每一步优化（30%-100% 提升）都揭示了一个硬件特性，从内存合并到 SMEM 到寄存器级协作。

> **验证说明**：本文所有数值计算均可通过配套 Python 脚本 `01-siboehm-cuda-matmul-worklog.py` 复现。脚本使用 `numpy.random.seed(42)` 生成相同的矩阵，并逐步验证每个 Kernel Stage 的输出与 numpy `@` 运算符的结果一致（最大误差 < 10⁻⁴）。
