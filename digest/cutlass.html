CUTLASS：把 SGEMM 优化经验做成模板

NVIDIA 这篇文章讲的是高性能 GEMM 的“工程化抽象”。如果前两篇让你知道每一步优化怎么写，CUTLASS 文章则告诉你：这些套路可以被拆成层级和模板参数。

原始网页：CUTLASS: Fast Linear Algebra in CUDA C++

原文线索

原文主题：CUTLASS 是 CUDA C++ 模板集合，用来把高性能 GEMM 的组成部分抽象成可组合、可调参的模板类。
关键概念：Thread Block Tile → Warp Tile → Thread Tile；数据从 Global Memory 到 Shared Memory，再到 Register File，最后进入 CUDA cores 计算。
CUTLASS 不是“又一种矩阵乘法公式”。它更像一套可调尺寸的积木：你决定每层 tile 多大、是否双缓存、输出最后要不要融合激活函数。

原文结构脉络

背景GEMM 是深度学习中大量计算的核心，cuBLAS 和 cuDNN 都依赖高性能矩阵乘法策略。
层级结构文章先解释 C tile、A/B tile、K 方向分块，再引出 thread block / warp / thread 的层级。
模板接口用 block task、policy 和 epilogue 等类型，把 tile 大小、布局、写回操作表达成 C++ 模板。
扩展能力通过 epilogue 融合 bias、ReLU 等后处理，减少额外 kernel 和 GMEM 往返。
性能结果最后展示 CUTLASS 在 V100 上接近 cuBLAS 的性能，说明抽象没有明显牺牲效率。
CUTLASS 的文章读起来比前两篇抽象，因为它不再只讲“一个 kernel 怎么优化”，而是在讲“怎样把很多高性能 kernel 的共同结构抽出来”。把它放在最后读，会更容易。

为什么需要层级

一个大矩阵无法直接全部放进高速内存，所以必须分层切块。每一层都回答一个不同尺度的问题。

Thread Block Tile一个 CUDA block 负责 C 的一块大矩形，并把对应 A/B tile 搬进 SMEM。
Warp Tileblock 内多个 warp 各自负责 C 子区域，减少互相干扰。
Thread Tile单个线程负责更小的 C 子块，把 accumulator 放进寄存器。
Instruction Tile最终映射到具体指令，比如 FMA、IDP4A 或 WMMA fragment。
这四层不是为了复杂而复杂。它们刚好对应 GPU 的硬件层级：block 对 SMEM，warp 对 SIMT 执行，thread 对寄存器，instruction 对 CUDA core 或 tensor core。

把 CUTLASS 参数翻译成人话

原文里的 policy 参数看起来很模板化，但含义可以拆开理解。

BlockItemsY / BlockItemsX: 一个 block 算多大的 C tile
BlockItemsK: 每轮 K 方向搬多厚
ThreadItemsY / ThreadItemsX: 一个线程算多大的 C 小块
UseDoubleScratchTiles: Shared Memory 是否双缓存
Epilogue: C 写回前还要做什么额外操作
当你在手写 kernel 里调 BM、BN、BK、TM、TN，本质上就是在做 CUTLASS policy 的低配版。CUTLASS 把这些选择类型化、模板化，让你能系统地生成很多变体。

双缓存和 epilogue 是两个工程重点

双缓存

一块 SMEM 给当前计算用，另一块 SMEM 预取下一轮数据。目标是让搬运和计算重叠。

Epilogue

矩阵乘完以后，写回 C 之前顺手做缩放、bias、ReLU 等操作，减少额外 kernel 和 GMEM 往返。

双缓存像两个托盘：厨师炒第一盘时，助手已经把第二盘材料摆好。Epilogue 像出锅前顺手撒盐、加酱，不用再把菜端回厨房加工一次。

如何把这篇和前两篇连起来

  1. Boehm 文章让你知道每一种优化为什么能快。
  2. wangzyon 仓库让你看到这些优化在 CUDA kernel 里怎么落地。
  3. CUTLASS 文章把这些经验整理成层级化模板，方便系统调参和复用。
所以学习路线应该是：先手写理解，再看仓库验证，最后用 CUTLASS 的层级抽象整理脑子。反过来直接读 CUTLASS，很容易只看到一堆模板参数，却不知道它们为什么存在。
