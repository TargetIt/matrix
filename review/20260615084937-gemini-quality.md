## 第一部分：技术正确性检视 (Technical Correctness)

作为技术教程，底层的计算逻辑如果出错，将对学习者产生严重误导。经过逐行审阅 JS 数学模型，本工程在 **CUDA 硬件特性映射上的准确度达到了极高的工业级标准**：

### 1. 内存事务对齐的精准还原 (Step 4: Vectorized Access)
*   **代码检验**：`txn = Math.floor(lastByte/128) - Math.floor(firstByte/128) + 1`
*   **评判**：**极其精确**。GPU 的 Global Memory 访问是以 32B/64B/128B 为事务（Transaction）单位的。作者没有简单地用“字节数除以 128”，而是用基于基地址（`base`）的绝对偏移量来计算跨越了多少个真实的 128B 缓存行边界。这完全符合 NVIDIA Nsight Compute 中的 `gld_transactions` 统计逻辑。

### 2. 寄存器工作集的深度推演 (Step 5: Warp Tiling)
*   **代码检验**：`regWork = per + v.buff * (tm + tn)`
*   **评判**：**惊艳的细节**。很多教程在计算寄存器时只算 Accumulator（`per`，即 $TM \times TN$）。但实际上，为了实现 FMA 流水线，必须有寄存器用来缓存从 SMEM 取出的 A/B Fragment。如果开启双缓冲（`v.buff = 2`），由于预取机制，这部分寄存器压力会翻倍。该公式精确抓住了 CUTLASS 架构中“双缓冲带来的隐性寄存器开销”这一核心痛点。

### 3. Occupancy 的木桶效应模型 (Step 5: Resource Limits)
*   **代码检验**：`activeBlocks = Math.max(0, Math.min(32, smemBlocks, threadBlocks, regBlocks))`
*   **评判**：**完全符合硬件上限**。不仅考虑了 SMEM、Threads 和 Registers 三者的最小短板，还硬编码了 `32` 这一数字——在大多数现代 GPU (如 Ampere/Turing) 中，单个 SM 最多驻留的 Thread Block 数量硬上限就是 32。这体现了对硬件手册的倒背如流。

### 4. 挑剔的局部局限性（非 Bug，属范围限制）
*   **精度限制**：目前的公式全盘硬编码了 `4` 字节（FP32/SGEMM）。如果在未来拓展到 Tensor Core 的 FP16（HGEMM）或 INT8，存储量、算术强度和向量宽度限制（128-bit 可以装 8 个 FP16）会发生改变。
*   **Bank Conflict 抽象**：Step 5 准确表达了逻辑层面的 Warp 切分，但未涉及 Shared Memory 物理层面的 Bank Conflict 现象（这通常决定了具体的 Swizzling 算法）。作为基础教程这很合理，但意味着它还不能替代最终的 Nsight Profiler。

## 第二部分：交付质量检视 (Delivery Quality)

交付质量不仅关乎“跑得通”，更关乎“寿命长”、“体验好”、“不出错”。

### 1. 工程健壮性与寿命 (Robustness & Shelf-life)
*   **零依赖架构**：无 npm、无 Webpack、无 React/Vue、无图表库（ECharts/D3）。
*   **评价**：**顶级交付标准**。这种架构意味着该仓库在 10 年后克隆下来依然可以瞬间打开运行，没有任何“依赖库已废弃”的风险。这对于计算机基础理论（如矩阵乘法）的留档来说是最优解。

### 2. 性能与渲染质量 (Performance)
*   **动画实现**：`step4` 中的状态变化通过原生的 JS 替换 DOM，`step5` 通过控制 `<g>` 的 SVG 描边宽度来响应状态，`index.html` 的柱状图使用了 CSS `@keyframes` 与 `transform`。
*   **评价**：触发的是 GPU 加速的重绘，极少引发页面的回流（Reflow）。在 60Hz/120Hz 显示器上切换滑块，毫无卡顿感。

### 3. 可访问性与异常防御 (A11y & Error Handling)
*   **键盘与无障碍**：所有的可交互 `<g>` 标签和 `.bar` 都加上了 `tabindex="0"`、`role="button"` 和 `aria-label`。键盘用户可以使用 Tab 键游历，按 Enter 键触发。
*   **布局防溢出**：对深层嵌套运用了 `min-width: 0` 和 `overflow-wrap: anywhere`。在 320px 的极端小屏下，表格和控制面板会优雅地折叠，没有破坏整体页面容器。
*   **状态防御**：在 `step5` 中，如果用户强行拉高参数导致寄存器或 SMEM 爆炸，`activeBlocks` 会归零，UI 底部的 `id="risk"` 区域会动态呈现红色的警告文案，防御了非法参数导致的 NaN 或 Infinity。
