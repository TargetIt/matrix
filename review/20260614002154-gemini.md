### 一、问题发现 (Findings)

#### 🚨 High (高优先级)
*   **问题描述**：在 `cuda-matmul-worklog/index.html` 的 "Occupancy 资源计算器" 中，Shared Memory (SMEM) 占用限制的计算遗漏了原文中明确指出的 CUDA Runtime 运行时开销。
*   **证据/路径**：`cuda-matmul-worklog/index.html` 第 64 行：
    `sb=smem?Math.floor(102400/smem):32`。而在其对应的 `original.md` 中有明确计算公式："Shared memory: 8192B/Block + 1024B/Block for CUDA runtime usage = 9216B/Block. (102400B per SM) / (9216B per Block) = 11.11 ⇒ 11 Blocks upper limit."
*   **影响**：违反了需求 6（“必须依据原文中的真实数据，不能使用无依据数据”）及内容正确性要求。用户滑动滑块到 8192B 时，可视化将错误显示可加载 12 个 Block，而原文/实际情况为 11 个。
*   **修复建议**：将第 64 行的相关公式修改为包含 1024B overhead 的逻辑：`sb=Math.floor(102400/(smem+1024))`，并相应更新 `#formula` 的文本展示。

#### ⚠️ Low (低优先级)
*   **问题描述**：`sgemm-practice/index.html` 中的 SVG 柱状图在切换不同内核优化阶段时，仅改变了选中柱子的填充颜色（`fill`），柱子的高度虽然反映了真实数据，但缺乏直观的动态补间动画（Transitions）。
*   **证据/路径**：`sgemm-practice/index.html` 第 47 行 `drawChart()` 每次被调用时均是直接全量替换 `svg.innerHTML`。
*   **影响**：尽管没有伪交互，但未能在视觉表达上最大化 SKILL 中提到的 "交互 > 静态" 与动画连贯性（类似 GSAP 带来的流畅体验）。
*   **修复建议**：由于未使用外部依赖，建议为 `<rect>` 添加基础的 CSS `transition: fill 0.3s, height 0.3s;`，或者通过 `requestAnimationFrame` 微调以增加页面高级感。
