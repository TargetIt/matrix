前言 GPU 内存层级（GMEM → SMEM → Registers）与性能上限

前言：为什么 SGEMM 优化几乎等价于“数据搬运优化”？

从公式上看，SGEMM 只是 C = A × B。但真正决定性能的，并不是乘加公式本身，而是数据从哪里来、被重复用了多少次。 NVIDIA GPU 上，单个 FMA 指令的执行速度很高；真正昂贵的，是一次又一次从Global Memory (GMEM) 把同样的数据重新读进来。

这就是为什么三层内存层级必须先吃透：GMEM 容量大但延迟高，适合存放完整矩阵；Shared Memory (SMEM) 更靠近 SM，适合一个 thread block 共享 tile； Registers 最快，适合每个线程私有的局部累加器。一个高性能 GEMM kernel 的本质，就是设法把数据使用路径压缩成： GMEM 读一次 → SMEM 重用多次 → Registers 再重用多次。

请先打开“前言演示”，拖动 Tile Reuse 滑块。你会看到：当同一批数据在 SMEM 和寄存器中被复用更多次时， 每次 FMA 平摊到的 GMEM 字节数会快速下降。这就是后续一切优化的主线。

从 CUTLASS 的角度看，这件事后来会被表达为层级化 Tile：Threadblock 负责从 GMEM 取大块数据，Warp 负责在 block 内继续分工，Thread/Instruction 负责寄存器级微内核。 但在进入那套抽象之前，先建立一个简单心智模型：越往内层，数据越贵重，越要尽可能“榨干”复用。

建议你在演示中重点观察的现象

  • 当复用率很低时，GMEM 条目会变成瓶颈，吞吐被“喂不饱”。
  • 当同一 tile 在 SMEM 中被多个线程反复消费时，GMEM 的压力下降最快。
  • 寄存器层的累加器让每个线程不必在每次乘加后都把中间结果写回慢速内存。
核心直觉：
1. GMEM 适合存“大而全”的原始矩阵。
2. SMEM 适合存“当前 block 正在吃”的 A/B tile。
3. Registers 适合存“当前线程私有”的 C 子块累加器。
4. 优化目标不是只减少指令数，而是减少“昂贵数据路径”的重复次数。

步骤 1：Naive SGEMM —— 正确，但几乎注定慢

朴素版本通常这样映射：一个线程负责矩阵 C 中的一个元素，线程先通过 (row, col) 定位自己的输出，随后在 k = 0..K-1 的循环中不断执行： 读取 A[row, k]、读取 B[k, col]、做一次乘加。 

sum = 0
for (k = 0; k < K; ++k) {
  sum += A[row, k] * B[k, col];
}
C[row, col] = sum;

这个 kernel 有两个优点：实现直接、逻辑清晰。但性能痛点也非常集中：

  • 每个线程在 k 循环中反复从 GMEM 读取 A/B，几乎没有缓存重用。
  • 相邻线程虽然都需要相近的数据，但它们没有显式协作，导致同一元素被多个线程重复搬运。
  • 算术强度低：一次 FMA 只配套少量计算，却伴随两次慢速读取。
请打开“Naive SGEMM 演示”，选择一个 C[row, col]，再点击 播放 k 循环。注意每走一步， 只为了当前输出元素的下一次乘加，就又发生了两次 GMEM 读取。你会直观看到： 线程是在算，但 GPU 更多时候是在等数据。

Simon Böhm 的路线之所以从 naive kernel 起步，是因为它把问题暴露得非常干净：一切后续优化，都可以理解为“让这段 k 循环需要的新数据更少、更快、更规则”。 

Naive SGEMM 的性能症结：
- 正确性没有问题。
- 并行度表面上也足够。
- 但数据复用几乎为 0。
- 所以瓶颈常常不是 ALU，而是 GMEM 带宽与延迟。

步骤 2：Global Memory Coalescing —— 先把 32 个线程“排整齐”

进入 CUDA 优化的第一道真正门槛，就是要理解：warp 中 32 个线程的全局内存请求，不是 32 次完全独立的小请求。硬件会尝试把它们合并成更少的事务； 对单精度 float 来说，如果 32 个线程恰好读取 32 个连续的 4B 元素，那么总请求正好是 128B，这通常可以映射成非常理想的合并事务。

问题在于：如果线程访问是跨列跳跃的，比如线程 0 读地址 0、线程 1 读地址 32、线程 2 读地址 64……那么这 32 个线程虽然都在工作， 但它们会落到许多不同的 128B 段里，结果是硬件不得不发起更多事务，传输了更多并不真正需要的字节。

请打开“Coalescing 演示”，把模式在 连续访问 与 跨步访问 之间切换，并拖动 Stride 滑块。 仔细观察右侧 128B 段的高亮数量：线程 ID 到地址的映射一旦变稀疏，事务数就会立刻膨胀。

这也是为什么很多 SGEMM 教程会先想办法让线程优先沿着内存连续方向读取，或者通过布局变换/转置/打包，让 warp 的 lane 地址更容易收敛到连续区间。 在后续 block tiling 里，这种思想会进一步演化成：先合并把 tile 拉进来，再在片上内存里慢慢切分与复用。

判断一个 GMEM 访问模式是否健康的三个问题

  • 32 个线程是否在访问尽量连续的地址？
  • 这些地址是否对齐到较少的 128B 段？
  • 硬件实际搬运的字节中，有多少是真正被线程用到的？
关键结论：
- 合并访存不是“可选优化”，而是后续优化的地基。
- 同样读取 32 个 float，理想情况只需 1 个 128B 事务。
- 一旦 stride 变大，事务数和浪费字节都会飙升。
- 访问模式的规则性，往往比单线程代码看起来“简洁”更重要。

步骤 3：Shared Memory Caching —— 让一个 tile 被整个 block 重用

合并访存解决了“怎么把数据高效搬进来”，但没有解决“同一数据被不同线程重复搬运”的问题。 Block tiling 的做法是让 thread block 协作加载 A 的一块行 tile 和 B 的一块列 tile， 先写入片上的 Shared Memory，再由 block 内所有线程重复读取。 

As[threadRow][threadCol] = A[globalRow][globalK];
Bs[threadRow][threadCol] = B[globalK][globalCol];
__syncthreads();

for (int k = 0; k < TILE; ++k)
  sum += As[threadRow][k] * Bs[k][threadCol];

__syncthreads();
打开步骤 3 演示，按“下一阶段”依次观察协作加载、同步、SMEM 复用和写回。 再切换到右下边界 tile，确认无效线程必须补零读取并禁止越界写回。
  • 第一次同步保证所有 As/Bs 元素写完后才开始计算。
  • 第二次同步防止较快线程提前覆盖下一轮 tile。
  • tile 越大，输入复用越高，但每个 block 的 SMEM 占用也越大。
从 Naive 到 Tiled：
- Naive：每个 C 元素独立读取整行 A 与整列 B。
- Tiled：每个 A/B tile 从 GMEM 读取一次。
- 同一输入在 block 内被多个输出线程消费。
- 代价：同步、SMEM 容量、边界分支与 bank conflict。

步骤 4：Vectorized Memory Access —— 用一条指令搬运连续的 float4

当地址连续且满足对齐约束时，float4 可以用一条 128-bit 指令搬运四个 FP32。 它不会减少矩阵必须传输的总字节数，但会减少 load/store 指令、地址计算和在途内存操作数量。 在原始 SGEMM 实践参数 BM=128、BK=8、256 threads 中， 每线程标量搬运需要四条指令，而 float4 只需一条。 

float4 v = reinterpret_cast<float4*>(&A[index])[0];

// 16B 对齐是 128-bit 访问的前提。
// 任意矩阵尺寸还需要前导或尾部标量路径。
在步骤 4 演示中改变起始偏移和元素数量。完全对齐时，指令数接近标量方案的四分之一； 偏移或长度不整除时，会自动出现前导和尾部标量访问。
  • 向量宽度与地址对齐必须匹配，float4 要求 16B 对齐。
  • 向量化和 coalescing 是两个维度：前者减少指令，后者减少内存事务浪费。
  • 真实 kernel 还要处理矩阵尾部、leading dimension 和转置布局。
收益来源：
- 更少的 LDG/STG 指令。
- 更少的索引计算。
- 更少的在途内存操作。

不能忽略：
- 对齐承诺。
- 非整除尾部。
- 128B segment 边界。
- SMEM 布局和 bank conflict。

步骤 5：Warp Tiling —— 用寄存器微内核完成层级化复用

Block tiling 之后，单线程只计算一个 C 元素会让 SMEM load 指令过多。 Warp tiling 进一步把 block 的 C tile 分给多个 warp，再让每个线程在寄存器中保存一个小型 accumulator tile。 A/B fragment 从 SMEM 进入寄存器后，会参与多个 outer-product FMA。

CUTLASS 将这套结构明确表达为 Threadblock Tile → Warp Tile → Thread/Instruction Tile。 tile 越大，数据复用和延迟隐藏机会越多；但 SMEM、寄存器和线程数也会上升，最终会压低 occupancy。

在步骤 5 演示中点击三个层级并调整 block、warp、K tile 与单双缓冲。 注意观察：双缓冲让 GMEM→SMEM、SMEM→RF 和 RF→FMA 重叠，但片上存储需求翻倍。
  • 同一行 warp 复用 A fragment，同一列 warp 复用 B fragment。
  • 线程 accumulator 越大，FMA/SMEM-load 比越高，但寄存器压力越大。
  • 高性能 policy 必须同时满足线程数、SMEM、寄存器和目标 GPU 架构约束。
层级数据路径：
GMEM
  ↓ coalesced / vectorized load
Threadblock Tile in SMEM
  ↓ warp fragment load
Warp Tile
  ↓ thread-owned fragment
Register Accumulator
  ↓ FMA / MMA
