# 需求与策略基线

## 原始需求

1. 从 NotebookLM 的 `MATRIX 演进跟踪` 笔记本取得三篇以 `[高]` 开头的来源。
2. 每篇来源建立独立子目录，先保留原文和来源证据，再提供交互可视化。
3. 使用 interactive-visualization Skill 的方式方法实现，并把工程放在 `matrix` 仓库。
4. 后续增加 CUDA SGEMM 前言和步骤 1 至步骤 5，形成由浅入深的学习路径。

## 决策框架

- Q1 受众：GPU/CUDA 学习者和芯片、性能工程师。
- Q2 目标：先教学理解，再允许参数探索和来源核对。
- Q3 内容：内存层级、地址访问、GEMM tile 层级、性能数据和资源约束。
- Q4 数据：NotebookLM 导出、上游原文实测数据，以及有明确公式和边界的浏览器端推导。
- Q5 资源：静态 HTML、CSS、JavaScript；优先可离线打开和低部署成本。

项目采用轻量静态网页，而不是引入 Svelte/D3。原因是当前数据规模小、没有后端，
部署目标是可直接浏览。共享计算必须放入 `assets/`，不得继续复制到多个页面。

## 数据等级

页面中的数字和视觉编码必须属于以下一种，并在附近明确说明：

1. `实测`：来源原文给出的测试结果。
2. `原文推导`：原文提供公式或完整计算过程。
3. `模型估算`：项目公式计算，必须说明架构、假设和适用边界。
4. `教学示意`：只表达关系，不得显示成百分比、吞吐或硬件实测值。

不得用主观填写的百分比、随机大小或装饰性长度冒充技术数据。

## 追踪关系

| 需求 | 交付位置 | 验收方式 |
|---|---|---|
| 三篇来源和原文 | 三个来源子目录的 `SOURCE.md`、`notebooklm-export.md`、`original.md` | 来源清单和 SHA-256 |
| 三个独立可视化 | 三个来源子目录的 `index.html` | 页面回归和来源核对 |
| 教程学习路径 | 根入口与 `demos/` | 导航、交互、移动端检查 |
| 数据真实性 | 页面文案、`assets/gemm-policy.js` | 模型单测和静态检查 |
| 合法 CUTLASS policy | 来源专题和 Step 5 | 32-lane、整除和资源断言 |
| 一键回归 | `scripts/test.sh` | 命令退出码为 0 |

## 视觉语义

全项目使用 `assets/semantic-colors.css`：

- 红色：GMEM 或错误风险。
- 蓝色：SMEM 或片上共享数据。
- 绿色：寄存器、线程私有数据或有效计算状态。
- 紫色：Warp。
- 橙色：数学运算、流水或警告。

颜色不得作为唯一信息载体，必须同时提供文字、数值或形状。
